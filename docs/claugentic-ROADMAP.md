## What this app is & does  _(generated by /claugentic-dev-harness:audit · do not edit — re-run to refresh)_

<!-- harness-audit:overview:start -->
_Inferred from project structure, manifests, and docs — not from running the app._

**What it is** — DistrictSync is a desktop/server **data-conversion tool** for school districts in British Columbia. It takes the raw student-information exports a district pulls from MyEducation BC and reshapes them into the specific CSV files that SpacesEDU / myBlueprint+ need to set up classes and rosters. It ships as a single double-clickable program (built with PyInstaller) so non-technical district staff can run it on a school server, often on an automatic daily schedule.

**What it does** — It reads up to a handful of district export files, cleans and reorganizes them, and writes up to seven output spreadsheets (students, staff, families, classes, enrollments, plus two optional course-catalog files). It can preview changes without writing, compare against the previous run, produce a data-quality report, and optionally deliver the finished files to SpacesEDU automatically over a secure (SFTP) connection. Each district's quirks (different file names, columns, email formats, calendar dates) are handled by per-district configuration rather than code changes.

**How it's built** — Python 3.13, using pandas for the data work and pydantic to validate the district configuration files. It's a classic ETL pipeline: **Extract** (read the export files) → **Transform** (entity-by-entity reshaping) → **Load** (write the CSVs). There's also a native desktop app (Flet) for staff who'd rather not use the command line, and built-in scheduling helpers for Windows and Linux.

**How it's organized** — The code lives under `src/`: the pipeline stages in `src/etl/` (with one focused transformer per output type under `transformers/`), configuration handling in `src/config/`, secure file delivery in `src/sftp/`, the scheduling helpers in `src/scheduler/`, shared safety/validation utilities in `src/utils/`, and the native UI in `src/ui_flet/`. Every district's mapping rules are plain YAML files in `config/mappings/`.

**Safety-net signals** — Strong. A large automated test suite (600+ tests) with an 80%-coverage gate, a "golden file" regression test that catches any accidental change to one district's output, configuration validation, and a full continuous-integration pipeline (lint, formatting, type-checking, a security scan, and the tests). The codebase also documents an explicit set of guarantees (atomic file writes, "no orphan" roster integrity, fail-loud error handling) and a clear exit-code contract.

**Confidence & caveats** — High confidence in the structural picture: this is a mature, well-documented, well-tested project with clear conventions. Because this overview is read from the code's structure (not from running the tool against real data), the audit that follows focuses on where the *standards* — especially **student-data privacy, data integrity, and reliable delivery** — could still be strengthened.
