## What this app is & does  _(generated by /claugentic-dev-harness:audit · do not edit — re-run to refresh)_

<!-- harness-audit:overview:start -->
_Inferred from project structure, manifests, and docs — not from running the app._

**What it is** — DistrictSync is a desktop/server **data-conversion tool** for school districts in British Columbia. It takes the raw student-information exports a district pulls from MyEducation BC and reshapes them into the CSV files SpacesEDU / myBlueprint+ need for rosters, classes, and course history. It ships as a single double-clickable program (built with PyInstaller) so non-technical district staff can run it on a school server — usually on an automatic nightly schedule set up by the built-in first-run wizard.

**What it does** — It reads a district's export files, cleans and reorganizes them, and writes up to eight output spreadsheets (students, staff, families, classes, enrollments, two optional course-catalog files, and an optional attendance feed). It can preview changes without writing, compare against the previous run, produce a data-quality report, and deliver the finished files to SpacesEDU automatically over a secure (SFTP) connection with the server's identity pinned in advance. Eleven per-district/tier configuration files handle each district's quirks (file names, columns, email formats, calendar dates) without code changes.

**How it's built** — Python 3.13, using pandas for the data work and pydantic to validate configuration. It's a classic ETL pipeline: **Extract** (read the export files) → **Transform** (one focused transformer per output type) → **Load** (atomic CSV writes). A native desktop app (Flet) provides the first-run setup wizard, a health dashboard, manual conversion, and run history backed by a small local database; Windows/Linux scheduling helpers automate the nightly run (with per-operation elevation on Windows, so the app itself never runs as administrator).

**How it's organized** — The code lives under `src/`: the pipeline stages in `src/etl/` (one focused transformer per output type under `transformers/`), configuration handling in `src/config/`, secure file delivery in `src/sftp/`, scheduling in `src/scheduler/`, run history in `src/history/`, shared safety/validation utilities in `src/utils/`, and the native UI in `src/ui_flet/`. Every district's mapping rules are plain YAML files in `config/mappings/`.

**Safety-net signals** — Strong. A large automated test suite (~1,686 tests) with an 80%-coverage gate, a "golden file" regression test pinning one district's exact output bytes, an output-schema contract test across every district config, configuration validation for all 11 configs, and a full continuous-integration pipeline (lint, formatting, type-checking, a security scan, and the tests). The codebase documents explicit guarantees: atomic writes with rollback, "no orphan" roster integrity, fail-loud error handling, a fixed exit-code contract, and a privacy split that keeps student data out of logs.

**Confidence & caveats** — High confidence in the structural picture: this is a mature, well-documented, well-tested project with clear conventions. Because this overview is read from the code's structure (not from running the tool against real data), the audit that follows focuses on where the *standards* — especially **student-data privacy, data integrity, and reliable delivery** — could still be strengthened.

**Deferred — triaged, with the reason each is not being done now.** Grouped by theme; every one is verified-real, so these are *judgment* deferrals, not dismissals.

- **Correctness, latent (no known partner hits it yet).** Ambiguous `dd/mm` vs `mm/dd` slash dates resolve to US order (`dates.py:30`) — MyEd BC exports `dd-MMM-yyyy`, so this bites only a district whose export changes shape; fixing it properly means a per-district `date_format` rather than a reorder that just flips who breaks. Accented names collide in generated emails (`sanitize` strips to `[a-z0-9]`, so `José`→`jos`) — affects SD60 only, and the fix **changes that district's output bytes**, so it wants its own snapshot-evidenced slice. Excel-resaved input (UTF-16) reads as garbage; a BOM sticks to the first column on the fallback parse path.
- **Unattended-run policy (owner's call, not an engineering default).** A collapsed roster is write-gated in Convert but written *and uploaded* by the nightly run. Auto-blocking delivery would also block a legitimate large drop (year-end), so the threshold and the opt-in/opt-out default are a product decision — highest blast radius on this list, and the one I most want an owner ruling on.
- **Test-suite integrity (real, but it protects us, not partners).** The isolation canary watches the abandoned `~/.districtsync` location, so it no longer proves tests leave the real user profile alone; the exit-code-3 test re-implements the logic and effectively asserts `sys.exit(3)` exits 3; there is no Windows CI job at all, so every win32-gated scheduler/elevation test never runs in CI on the product's main platform.
- **Robustness under bad conditions.** Schedule registration and cron have no subprocess timeout (Setup can hang with both buttons greyed out) while the read-back beside them is capped at 10s; SFTP bounds connect but not transfer, so a half-open socket latches the UI job state permanently; the roster-shrink baseline counts *file lines*, so a quoted newline in an address invents a false 30% drop alarm.
- **Ledger accuracy.** "Most recent run" sorts on naive local-time text, so an NTP correction or the DST fall-back can reorder history and show yesterday's green over today's failure; cron runs are tagged `cli` instead of `scheduled`; manual conversions reach the run store but never the diagnostic log.
- **Convert/Deliver edges.** Deliver names the zip after the *currently selected* district regardless of which district built the files on disk; Convert matches input filenames case-sensitively while the CLI does not, so the same setup can work nightly and produce nothing on click; navigating away from Convert and back re-arms the button mid-run.
- **Structure & docs (no partner-visible symptom).** ~12 maintainability items (the run record is an untyped dict read by string literal across six modules; two hand-synchronised copies of the run-a-conversion sequence that have already diverged; `mypy` disabled for the whole UI package including 17 modules that never import Flet; ruff still targeting py39 on a py313 project) and ~8 further doc-accuracy defects beyond the six fixed here (the developer guides describe a context field and config formats that don't exist; the attendance feed ships in no guide; the version number disagrees across four places).
- **Product polish.** Six `product-ux` items, all T3 — multiple filled primaries per screen against the design system's one, the schedule-attention dot having no accessible name, the onboarding subtitle failing AA where the automated contrast gate can't see it, wizard focus moving on only one of five steps.
