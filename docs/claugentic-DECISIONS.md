- **2026-06-19 — Plan 0004: StudentAttendance output reduced from 28 columns to the 4 required SpacesEDU fields.** The SpacesEDU docx explicitly permits dropping every optional field after Student Number ("Only required fields are needed in the file, all other fields can be removed if preferred"), so `StudentAttendance.csv` now emits ONLY the 4 populated columns in exact case-sensitive order: `School Number, Absence Date, Absence Category, Student Number`. Dropped the 24 always-blank optional columns. `OUTPUT_COLUMNS`/`POPULATED_COLUMNS` consolidated to one constant; `_frame()` no longer reindexes to a 28-col superset with `fill_value=""` (rows already carry exactly the 4 keys). The base `myedbc` `StudentAttendance.field_map` was reduced to the 4 keys (field_map is used ONLY for output column ORDER here — the transformer builds rows directly and never calls `apply_field_map`, so the values stay placeholders). All band logic preserved (role-based resolution, fail-loud derivation, no `drop_duplicates`, blank-code/blank-student drops, `format_dd_mmm_yyyy`). SD51 row count unchanged (79,303); SD74 snapshot byte-identical (not enabled).
- **2026-06-19 — Plan 0004: StudentAttendance band selection moved to per-district `source_files` (role-based, order-independent).** The base `myedbc` entity previously forced BOTH bands by declaring `daily_absences` + `period_absences` in `StudentAttendance.source_files`; deep-merge can't *remove* an inherited `source_files` key, so a single-band district couldn't run only daily or only period. **Fix:** the base now declares **no** `source_files` for StudentAttendance (empty `{}`; it never enables the entity, so it never runs there) and keeps only the shared defaults all districts inherit — the two headerless `headers:` blocks, the 28-col `field_map`, and `global_config.attendance` with both `daily` and `period` sub-blocks. Each **district** declares the role(s) for the band(s) it runs (`daily_absences` K-7 and/or `period_absences` 8-12); SD51 declares both (daily first → stays the pipeline primary). The transformer now resolves **each band by its ROLE** from `context.raw_data` (via `get_source_file`) instead of treating the positional primary `df` as the daily frame — so a **period-only** district (whose period file is primary) is no longer mis-processed as K-7 daily. Each band is optional/graceful and each band's config block is required **only when that band's data is present** (`attendance.daily` gated on the daily frame, `attendance.period` on the period frame), so a single-band district need not configure the other. **SD51 output byte-identical** (it declares both, daily first → `get_source_file(daily_absences)` returns the same frame the pipeline passed as `df`); **SD74 snapshot byte-identical** (not enabled). `sd51attendance` (attendance-only tier, `_base: sd51myedbc`) inherits both bands. No Pydantic change (`EntityConfig.source_files` already accepts an empty dict).
- **2026-06-17 — claugentic-dev-harness installed alongside the project's pre-existing in-house harness.** DistrictSync already maintains its own `docs/WORKFLOW.md`, `docs/ENGINEERING_STANDARDS.md`, `docs/DECISIONS.md`, `docs/ROADMAP.md`, `docs/ARCHITECTURE_TREE.md`, `.claude/agents/` roles, and a `scripts/check_architecture_tree.py` gate. The claugentic harness installs under the `claugentic-` namespace (its own tree + a second tree-gate hook, per the adopter's init choice) so the two coexist without collision. The harness **composes with** the project's existing CI gates (ruff / mypy / bandit / pytest 80% / SD74 snapshot / config-validation), it does not replace them. _(init harvest of `docs/WORKFLOW.md`.)_
- **2026-06-17 — Load-bearing project invariants the harness must respect** (harvested from `docs/WORKFLOW.md` + `docs/DECISIONS.md`, recorded so claugentic agents don't violate them): (1) **Config-driven columns** — GDE source column names come from the district `field_map`, never hardcoded in transformers; (2) **Zero-orphan invariant** — enrollment/class rows are filtered to `context.active_student_ids` so nothing references a student absent from `Students.csv`; (3) **`_base` config inheritance** — all 7 entity definitions live in base `myedbc_mapping.yaml`; configs select via `enabled_entities`, never redefine entities. See the project's own `docs/DECISIONS.md` for full detail.

- **2026-06-19 — Plan 0004 Slice 3: StudentAttendance 8-12 Student Period Absences band (per-period pass-through).** Added the 8-12 band as **per-period PASS-THROUGH** unioned with the existing K-7 daily band into the single 28-column `StudentAttendance.csv`. **One output row per period-absence row — no AM/PM collapse, no Period Id, no derivation** — because SpacesEDU aggregates per-day itself via per-entry weights (8-12 absence = 0.25/entry → 4 entries = 1 day, capped at 1/day); the standard rotating-block timetable export has no AM/PM signal anyway, so collapsing would be guesswork. **Category passed through AS-IS** (the GDE `Absence Category` already holds the final codes `A`/`A-E`/`L`/`AD`/`AL`/`OffSite`/`ISS`…; SpacesEDU ignores non-accepted ones, so no filtering). Rows with a blank category OR blank student number are dropped; **no dedup** (per-period multiplicity is intentional). Source = the **standard headerless** `StudentPeriodAbsences.txt` (MyEd Data Elements §2.45, 17 cols) — NOT the `Enhanced` variant the original plan assumed (Enhanced is headed + adds a Period Id the pass-through model doesn't need; standard is consistent with the headerless `StudentDailyAbsences.txt`). `daily_absences` stays FIRST in `source_files` so it remains the pipeline's primary (skip-on-empty-primary guard). New `global_config.attendance.period` block (4 source column names, no derivation knobs); reuses the open-dict `attendance` model from Slice 2 (no Pydantic change). Both bands optional/graceful (daily empty → period only; period absent → daily only). SD51 inherits the base source filenames unchanged (no per-district override). **SD74 snapshot byte-identical**; K-7 daily behavior unchanged. This supersedes the original Slice-3 plan (Enhanced-file half-day collapse), which was parked pending an Enhanced re-export.
- **2026-06-18 — Plan 0004 Slice 2: StudentAttendance entity scaffold + K-7 Daily band (SD51).** New opt-in `StudentAttendanceTransformer` emits the 28-column SpacesEDU `StudentAttendance.csv` (4 columns populated, 24 blank, **no `drop_duplicates`** — a full-day absence is intentionally two identical rows). **Derivation is fully config-resident** (Configurable-Columns + fail-loud): the K-7 `(Absent Code, Authorized) → Absence Category` map, the `Portion Absent → row-count` rule, and every source column name live in a new `global_config.attendance.daily` block read at runtime — so SpacesEDU's review tunes config, never code. An unmapped non-blank `(code, authorized)` pair **raises** rather than silently dropping/mis-bucketing. `attendance` was added to the `GlobalConfig` Pydantic model as an **open dict** (not a nested model) and threaded through `to_raw_dict()` so it reaches the transformer and so Slice 3's 8-12 Enhanced-Period keys need no model change. New `format_dd_mmm_yyyy` transform (added to `ALLOWED_TRANSFORMS`) produces the contract's `DD-MMM-YYYY` shape (mirrors `normalize_iso_date`). SD51 enables it via a **full** `enabled_entities` list (base-5 rostering + StudentAttendance) because deep-merge **replaces** lists. Quality report gets an explicit `StudentAttendance: []` skip (duplicates legitimate). **SD74 snapshot byte-identical** (not enabled for SD74); no other district affected. The 8-12 Enhanced Period Absences band is deferred to Slice 3 (pending SD51's `StudentPeriodAbsencesEnhanced.txt` re-export).

## Project history (harvested from the in-house harness, 2026-06-17)

## 2026-06-05 — Plan 0003 verification + import model (full-snapshot, upsert)

- **Verified end-to-end on real SD48 GDE data** (after rebasing onto the post-mbponly/extractor main): roster 7,922 → 7,543 (−379 = 50 inactive-without-withdraw-date leak + 329 PreReg, all verifiably non-active); orphaned student enrollments 405 → 0 on a full `--sis sd48myedbc` run (not just fixtures), stable under the rewritten extractor. SD74 snapshot byte-identical.
- **GDEs are full snapshots, not deltas.** Of 6,278 students in the SD48 schedule, all but 1 are present in the demographic extract — so the orphans were withdrawn/inactive students *in* the extract (removed by the active rule), not active students missing from a delta. The 1 exception (`2644905` — in the schedule, absent from the demographic) is a source-data gap → ROADMAP.
- **SpacesEDU import is upsert/unenroll-style** (`docs/partner/faq.md`): it preserves existing classes, adds new enrollments, and *unenrolls* students who drop out of `Enrollments.csv`. So dropping a withdrawn student's enrollment cleanly **unenrolls** them — which is *why* the zero-orphan filter is correct (the intended outcome), not merely "avoids import errors".

## 2026-06-05 — Zero-orphan enrollments (plan 0003, Slice 2)

- **Zero-orphan invariant via a published active roster.** `StudentTransformer` publishes its filtered output's `User ID` set to `context.active_student_ids` (so `active_student_ids == set(Students.User ID)` by construction); `classes.py` (homeroom) and `enrollments.py` (homeroom + subject) then filter their *student* rows to that set. Result: no enrollment or homeroom class references a student absent from `Students.csv`. Verified on an SD48-shaped fixture: 3 orphans (2 homeroom + 1 subject) → 0. The single filter helper is `BaseTransformer.filter_to_active(df, student_col, context, caller)` — both consumers call it (DRY); chosen over re-deriving "is active" at each read site (would drift from the roster). Demographic `Student Number` and schedule `Student ID` share the pupil-number value space; both sides normalized with `astype(str).str.strip()`.
- **Students-only filter; teacher/co-teacher rows untouched.** Only the student-row derivations are filtered; teacher rows in `_homeroom_enrollments`/`_subject_enrollments` derive from the *unfiltered* frames, and `_blended_teacher_enrollments`/`_classinfo_coteacher_enrollments` are not touched — so teacher rows are byte-identical to before (regression-tested by holding the homeroom class set constant and comparing a mixed roster vs an all-active control). The one intended teacher-row change: a homeroom with *only* inactive students is dropped (no empty homeroom class) along with its teacher enrollment — the no-empty-homeroom goal, not a leak.
- **Never filter-to-empty (guard).** When `active_student_ids` is empty (Students disabled / ran later, e.g. `mbp_core`) or the student column is absent, `filter_to_active` logs a WARNING and returns the frame unchanged rather than dropping every row. `filter_to_active` returns a `.copy()` so callers can mutate columns without a `SettingWithCopyWarning` (matches the other `filter_*` helpers).
- **SD74 snapshot byte-identical.** SD74 has no inactive students, so its active set = all students → the filter is a no-op; the frozen snapshot stays green (the regression guard).

## 2026-06-05 — Active-student predicate (plan 0003, Slice 1)

- **Config-driven active predicate (single source of truth).** Active-student detection now lives in `BaseTransformer` (`resolve_active_config` / `compute_enroll_status` / `is_active_mask`), resolving the status / withdraw-date columns and `active_values` from the Students `EnrollStatus` config (Configurable Columns). `students.py` delegates to it. Replaces the hardcoded-column logic in `students.py._determine_enrollment_status`. (Slice 2 will reuse the same mask to drop orphan enrollments.)
- **Corrected status-column alias.** The default status column resolves from the alias `["enrollment status", "enrolment status"]` — first present in the (normalized) frame. Real MyEd BC exports use the two-L `"Enrollment status"`; the repo fixtures / SD40's injected headers use the one-L `"Enrolment Status"`. The old code only matched one-L, so real two-L data silently fell through to withdraw-date-only detection (the SD48 −50 leak). A district can override `status_column` explicitly.
- **Status wins; withdraw date is the fallback.** When a row has a status value, the status decides (active iff in `active_values`) and the withdraw date is **ignored** — so a re-enrolled student (status `Active` with a lingering past withdraw date) stays Active. Only rows with *no* status value fall back to the withdraw date (past/unparseable → Inactive + one aggregated warning; 4 formats `%d-%b-%Y`, `%Y-%m-%d`, `%m/%d/%Y`, `%d/%m/%Y`). This matches the original status-XOR-date intent — corrected from a briefly-shipped "withdraw-date hard override" that wrongly dropped re-enrolled students (SD48: 3 such, roster 7,543 → 7,548).
- **PreReg excluded by default.** Default `active_values = ["Active"]`, matching `docs/partner/faq.md` ("Students with status PreReg … are excluded"). The active mask is `label != "Inactive"` (no implicit union with `"Active"`), so a district can opt PreReg (or any status) back in via `EnrollStatus.active_values` — e.g. for upcoming-year rosters. (An earlier draft of this plan retained PreReg; corrected after confirming the documented partner behavior against the SD48 data.)
- **`EnrollStatus` config fails loudly.** New strict `FieldEnrollStatus` (`extra="forbid"`) validates the dict form; a recognizable EnrollStatus override carrying an unknown/typo'd key now RAISES at config load (was a silent warn-passthrough). Bare `EnrollStatus: null` stays the auto-detect sentinel. SD48 roster correction: −50 verifiably-inactive students (Withdrawn/Inactive/Graduate without a withdraw date). SD74 snapshot stays byte-identical (its demographic has no status column + all-blank withdraw dates → date-only back-compat).
- **Deferred (ROADMAP):** `classify_field` is not idempotent — re-validating an `EntityConfig` that already holds typed field values (e.g. nested in a freshly-constructed `MappingConfig`) stringifies them. Pre-existing; out of scope for this slice. The real `load_config` path (raw dict → validation) is unaffected.

## 2026-06-05 — Plan 0002 Slice 4 (unattended SFTP + schedule hardening)

- **Scheduled task runs as the interactive setup user with stored Windows password** (`/RU /RP /RL HIGHEST`) so it survives reboots; SYSTEM rejected — SYSTEM cannot read the per-user OS keyring where the SFTP password is stored.
- **SFTP secrets stay in the per-user OS keyring; the task's run-as account must equal the setup account** — the Setup Wizard enforces this by resolving `current_run_as_user()` and registering the task as that account.
- **A requested SFTP upload that fails exits code 3** (ETL output CSVs still written), so delivery failures surface visibly in Task Scheduler's "Last Run Result" and in Run History, not silently as a green exit 0.

## 2026-06-04 — Harness / standards

- **Output targeting via `enabled_entities`.** All 7 entity definitions live in base `myedbc_mapping.yaml`; configs select which to emit via `global_config.enabled_entities`. Tiers `mbp_all` (7) and `mbp_core` (Students + 2 course CSVs) are thin overrides. Rationale: defines entity templates once (DRY) and lets a per-district myBlueprint+ config (`_base: <district>` + `enabled_entities`) inherit both the district's columns and the base entity defs. Chosen over "the new config carries the new sections" (would duplicate defs and hit YAML's single-`_base` limit). Ratified after PR #12 review.
- **`ARCHITECTURE_TREE.md` + presence enforcement.** Added `docs/ARCHITECTURE_TREE.md` (one line per source file). `scripts/check_architecture_tree.py` is a deterministic gate (no LLM) checking presence + staleness for `src/**/*.py` and `config/mappings/*.yaml`; descriptions are authored by humans/agents. Intended wiring: `make check-tree` + CI (authoritative) + optional local pre-commit.
- **Configurable-columns standard reaffirmed.** GDE source columns must be resolved from `field_map`, never hardcoded in transformer code. `student_courses.py` (PR #12) violates this (hardcodes ~10 columns, bypasses `field_map` for input) — logged as debt to migrate.
- **Line-ending hygiene (proposed).** Working tree shows widespread CRLF↔LF churn; recommend a committed `.gitattributes` (`* text=auto eol=lf`) + one normalization commit to stop the noise.
- **Adopted the staged agent Development Workflow** (`docs/WORKFLOW.md`): triage → discuss → plan (`.claude/plans/`) → adversarial plan-review → spec → user approval → implement (isolated) → verify → land/archive → retrospect. Roles are a *library* in `.claude/agents/` the orchestrator selects from per task (starter: `plan-reviewer`, `implementer-architect`). Hard rule: slice work into units one specialist finishes in a single ≤1M-context session, landing complete with **no new tech debt**. Stage 9 feeds learnings back into the harness (living-harness principle). Requires un-ignoring `.claude/agents/` + `.claude/plans/` so the role library and plans are shared (see `.gitignore`).
- **Architecture-improvement program → ROADMAP.** The full-codebase architect review (6-agent pass) findings are captured as a tiered backlog in `docs/ROADMAP.md`; each item runs through the Development Workflow in its own session rather than one large change.
- **Role library = local, version-controlled, shared; NO runtime injection.** Agent roles live in `.claude/agents/` (committed, grow over time, shared via the repo / packageable as a plugin later). We do **not** fetch role/skill text from online marketplaces at runtime and inject it into subagents — that's a prompt-injection / supply-chain / non-reproducibility risk. Marketplaces (incl. third-party aggregators like skills.sh / skillsmp.com) are **authoring-time discovery only**: review → vendor a pinned copy → commit. Prefer Anthropic first-party sources; treat aggregators as untrusted.
- **ARCHITECTURE_TREE enforced by hooks.** `.claude/settings.json` wires `scripts/check_architecture_tree.py --hook` as a `PostToolUse(Write)` nudge (prompts the agent to document a newly-created in-scope file, with a one-line description) + a `Stop` backstop (blocks finishing until the index is current). Script = deterministic presence/staleness gate; description authored by the agent that made the file.
