## Problem
Help is the **"I'm a non-technical admin, something's confusing / I'm stuck — where do I go?"** surface. The admin opens DistrictSync 2–3×/year; when they reach for Help they need, in one glance and ideally one click: *the canonical answers live here* (the org Help Centre), *a human I can email* (support), and a *plain reassurance* of what this tool even does and that closing the window doesn't stop the nightly sync. That's the whole job.

Today the Flet `help` destination is a **branded placeholder** (`build_placeholder`, `shell.py:117`); the real surface lives only in Streamlit `05_Help.py` (deleted at CUT-1). That Streamlit page is a **six-tab documentation browser** — it renders `partner/installation.md`, `partner/headless-sftp-setup.md`, `developer/architecture.md`, `partner/faq.md`, `partner/troubleshooting.md`, and four nested *developer* docs (setup/testing/adding-district/release). For the actual persona that is **over-built and off-target**: a non-technical admin does not read `developer/architecture.md` or `developer/release.md`; the canonical, always-current docs home is now the **org KB / SpacesEDU Help Centre** (per the 2026-06-30 MkDocs-removal decision and 0013's scope-lock), and the bundled `docs/` markdown is on a documented deprecation path (0013 config-capability epic step 4: "reconcile/remove the now-redundant bundled `docs/`"). Porting the six-tab browser faithfully would **re-entrench the very redundancy the program has decided to retire**, and would cost a `flet-pack.yml` bundling change + a bundle-path resolver to keep the bundled docs reachable in the frozen exe.

IA-7 ports this **minimal-but-complete, per the scope-lock**: a calm, branded Help surface that gets the admin un-stuck in one click — a prominent **"Open the Help Centre"** button (`page.launch_url` → the grepped canonical article), a clear **support-email** affordance, and a short plain-language **"what DistrictSync does / where the sync runs"** reassurance — consistent with the rest of the cockpit's verdict-first, humanized voice. It does **NOT** render the bundled `docs/` markdown (the decision below justifies why the link-out is the minimal-complete answer and the bundled-docs render is not proportionate).

## The bundled-docs-vs-link decision (the CENTRAL call)
**RECOMMENDATION: link out to the org KB + support email — do NOT render bundled `docs/` markdown.** This is the minimal-but-complete Help surface the scope-lock calls for. Justification (the tension between the IA-7 row and the scope-lock, resolved):

- **The scope-lock supersedes the row.** 0013 line 9 states the scope-lock block wins on conflict with the slice table; the block says IA-7 is "**minimal/flexible** (likely an org-KB link rather than rendering bundled markdown)". The row's "render `docs/` markdown natively" predates the docs-strategy shift and is explicitly the thing the block re-scopes. **The scope-lock is the authority (harness rule); the row is the stale artifact.**
- **The org KB is now the canonical docs home.** The 2026-06-30 MkDocs-removal decision retired the docs *site* precisely because the release body points users to the SpacesEDU Help Centre, and it defers the partner-facing docs strategy to the org-KB decision. The bundled `docs/` is explicitly slated for "reconcile/remove" in 0013's config-capability epic (step 4). **Rendering the bundled docs in-app would re-entrench redundancy the program has decided to retire** — the opposite of the scope-lock's intent.
- **The bundled render is disproportionate (cost the row's "M/low" hides).** It is NOT just an `ft.Markdown` call. To render bundled docs in the *frozen windowed exe* we'd need: (1) a `flet-pack.yml` `--add-data "docs${{ sep }}docs"` change (grep-confirmed absent today — only `config` is bundled); (2) a **bundle-path resolver** (`_MEIPASS`-aware, mirroring `launcher.resolve_frozen_cwd`) — a new COUNTED pure module + tests — since the dev path (`repo/docs/`) and the frozen path (`_MEIPASS/docs/`) differ; (3) a **curation decision** on WHICH of the 10 `docs/` files a non-technical admin should see (the six Streamlit tabs include `developer/architecture.md`, `developer/release.md` — plainly wrong for the persona); (4) markdown-render fidelity/theming work (the `docs/` markdown carries MkDocs frontmatter + admonitions the Streamlit page strips ad-hoc). That is real infrastructure + a curation call for content the KB already owns — **gold-plating an offline docs browser the scope-lock explicitly says not to build.**
- **The link-out is complete for the persona.** A 2–3×/yr non-technical admin who is stuck needs the canonical answers (the KB, always current, no version-skew with a frozen bundle) and a human (support email). Both are one click. The plain "what this does / where the sync runs" reassurance closes the "what even is this" gap in-app. Nothing the admin needs is missing.
- **`ft.Markdown` existing does NOT change the call.** The API is present on 0.85.3 (introspected below) — so this is a **scope decision, not a capability gap**. If the deferred docs-strategy decision later rules that a small curated offline subset should ship in-app (e.g. an offline FAQ/troubleshooting for an air-gapped district server with no browser), that is a clean, well-scoped ROADMAP follow-on: add `--add-data docs`, a COUNTED bundle-path resolver, a curated file list, and an `ft.Markdown` render. It is NOT this slice, and building it now would pre-empt a decision the program deliberately deferred.

**The gate rules.** If the gate overturns this and mandates the bundled render, the fallback is scoped in the [Decomposition](#decomposition-slices) note (a second slice: bundle-path resolver + `flet-pack.yml` + curated `ft.Markdown` render) — but the recommendation, grounded in the scope-lock + the docs-strategy decision + proportionality, is **link-out only**.
