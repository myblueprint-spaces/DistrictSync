## Problem
Run History is the admin's **read-only "is the sync actually working?"** surface: they open it 2–3×/yr (or after a scare) and want two things answered at a glance — *"has my nightly sync been running lately?"* (a staleness/verdict banner) and *"did each run actually work — build, deliver, no data problems?"* (a table of past runs, newest-first). It is the evidence that reinforces the trust the whole cockpit exists to give: Home shows the LATEST run's verdict; Run History shows the *history* — the proof the nightly job has a heartbeat.

Today the Flet `run_history` destination is a **branded placeholder** (`build_placeholder` in `shell.py`); the surface lives only in Streamlit `02→03_Run_History.py` (deleted at CUT-1). That Streamlit page inlines its own `__DISTRICTSYNC_RUN__` parser (now superseded by the landed `run_log.read_run_records`), renders a `pd.DataFrame` via `st.dataframe` (carrying the pyarrow-int64 gotcha its `_fmt` guards against), and — critically — **puts the raw `error` string and the resolved log path directly on screen** (the `Error` column at line 113, the log-path caption at line 136, the raw-log expander at 128-136). For a non-technical admin that is noise at best and a privacy leak at worst (the `error` is `str(e)` — it can carry an absolute path / `sis_type` / column name). The Streamlit page also leads with a table, not a verdict — it never answers "is my sync OK *right now*" before showing the log.

IA-6 ports this **verdict-first, humanized, privacy-safe**: a staleness/verdict banner at the top (reusing the landed `is_stale` + the latest-run verdict), then a plain-language `ft.DataTable` of the runs (friendly times, plain status labels, ✓/✗/— SFTP, a data-warnings count — **never a raw path/`error`/ISO**), with the three read-only degradation states each distinct and calm (empty `[]` → "no runs yet"; unreadable `None` → "history unavailable"; a malformed/partial record → a defensively-read row, never a crash). **Lower-risk than IA-4/IA-5** — no credentials, no worker thread, no ETL run: it only READS + displays a small local log.

## The derivation (the explicit, tested contract)

### `HistoryBanner` (the verdict-first banner state)
Frozen dataclass: `HistoryBanner(verdict: Verdict, headline: str, detail: str)`. `derive_history_banner(records, app_config, *, now=None)` — evaluated top-down, first-match-wins (mirrors `home_status`'s degradation-first order; assumes a configured install, same as Home — but Run History is reachable from the rail regardless, so it must also handle the pre-first-run states calmly):
1. **`records is None`** (unreadable log) → `WARNING`, headline *"Run history unavailable"*, detail *"We couldn't read the run log right now — your nightly sync may still be running normally."* (the calm never-crash floor; distinct from empty).
2. **`records == []`** (no runs yet) → `WARNING` (never red), headline *"No sync has run yet"*, detail *"Your nightly runs will appear here once the first one completes"* (+ if `app_config.schedule_registered`: *"— scheduled for {friendly schedule_time} each night."*).
3. **latest `status != "success"`** → `FAILED`, headline *"Your last sync failed"*, detail a **FIXED category sentence** *"The most recent run hit a problem and didn't finish — see the run below."* **[NEVER interpolate the raw `error`.]**
4. **latest success but `sftp_attempted and not sftp_ok`** → `FAILED`, headline *"Your last roster didn't reach SpacesEDU"*, detail *"The most recent run built the data but the upload failed."*
5. **latest success + delivered, but `is_stale(latest.timestamp, now)`** → `WARNING`, headline *"No recent sync"*, detail *"Your last sync was {friendly_timestamp} — a nightly run may have been missed."* (**the "your last sync was N days ago" the brief calls for**; reuses `is_stale` + `STALE_AFTER_HOURS`).
6. **latest success + delivered + anomalies non-empty** → `WARNING`, headline *"Something looked off recently"*, detail plain anomaly-count summary (never the raw `ANOMALY:` string).
7. **latest success + delivered + `data_errors.total > 0`** → `WARNING`, headline *"Recent runs completed with data warnings"*, detail *"Some records had field problems and were skipped — the runs still delivered."*
8. **else (recent clean delivered success)** → `HEALTHY`, headline *"Your sync is running"*, detail *"Your last sync delivered cleanly {friendly_timestamp}."*

*(Rules 3-8 mirror `home_status`'s precedence exactly — the DELIBERATE reuse of that vocabulary keeps Home + Run History consistent. The banner answers the same "is my sync OK?" question Home does, over the same latest record; the table below adds the history. If a `home_status` helper can be cleanly reused for the banner classification rather than restated, prefer that at Spec — decide whether to call a shared internal or restate the 6 rules; see design-question 4.)*

### `RunRow` (the per-run display model) + `SftpDelivery`
- `SftpDelivery` (Enum): `DELIVERED` (✓, `sftp_ok`), `FAILED` (✗, `sftp_attempted and not sftp_ok`), `NOT_ATTEMPTED` (—, else) — mirrors the Streamlit `"✅"/"❌"/"—"` ternary at line 112, but as a typed enum the view maps to a glyph/label (no raw emoji in the pure model).
- `RunRow` (frozen dataclass, PII-free): `when: str` (friendly_timestamp), `status_label: str` (a plain per-run label — see below), `status_verdict: Verdict` (drives an optional row colour cue), `entity_counts: dict[str,int]` (5 rostering always + 2 myBlueprint+ when-nonzero, reusing `home_status`'s entity tuples; `StudentAttendance` omitted), `entity_total: int`, `sftp: SftpDelivery`, `warnings: int` (`data_errors.total`), `duration: str` (plain "3.2s" / "—").
- `to_run_row(record, *, now=None)` — the total, defensive mapper. **`status_label`** re-derives the per-run label from the record's structured fields ONLY (the same precedence `_status_cell` uses — `status` → SFTP → data-errors — but in plain words, no emoji, no raw string): `"Failed"` / `"Built, not delivered"` / `"Delivered · N data warnings"` / `"Delivered"` / `"Completed"` (when SFTP not attempted). Reuse `home_status`'s classification vocabulary so the row label + the banner never contradict. Every field read is `.get` + `_as_int`-guarded (a partial/old record → a safe row: missing timestamp → `friendly_timestamp("")` → "recently"; missing counts → 0; missing status → treated as non-success → "Failed", the honest default).
- **Totality (TESTED):** `to_run_rows(records)` over any list — including records missing every key — returns a `list[RunRow]` with no raise; a `record` whose `error` carries a fake path (`C:\Users\x\secret`) produces a `RunRow` whose EVERY string field contains none of that substring (the privacy assertion).

## The `ft.DataTable` component (`components.run_table`) — first consumer
`ft.DataTable`/`ft.DataColumn`/`ft.DataRow`/`ft.DataCell` are the confirmed Flet 1.0 table controls (introspected against the installed `flet==0.85.3`; fields verified: `DataTable.columns/rows/heading_row_color/data_row_color/column_spacing/border/border_radius/divider_thickness/heading_text_style/data_text_style/show_bottom_border`; `DataColumn.label/numeric`; `DataRow.cells/color`; `DataCell.content`). A **coverage-omitted** factory in `components.py` (view glue, mirrors the other primitives):

- `run_table(rows: list[RunRow]) -> ft.Control` — builds the DS-1-styled `ft.DataTable`:
  - **Columns** (mirroring the Streamlit proven set, minus the raw `Error` column which is dropped for privacy): **When · Status · Students · Staff · Family · Classes · Enrollments · [Courses · Student courses when any row has them] · SFTP · Warnings · Duration**. Count columns use `ft.DataColumn(numeric=True)`. The 2 myBlueprint+ columns render ONLY if any row has a non-zero count (a SpacesEDU district shows the 5 rostering columns, not 7 with two all-zero columns — mirrors the tile rule in `home_status`/`home.py`).
  - **Cells:** every cell is a `ft.Text` of a uniform string (the "coerce to uniform string" lesson from `_fmt`, applied structurally in the pure `RunRow`); a not-produced entity renders `"—"`; SFTP renders the `SftpDelivery` glyph (✓/✗/—) with a non-colour label cue; Status renders `status_label` (optionally tinted by `status_verdict` via `DataRow.color` — decide at Spec, keeping AA contrast per `UI_CONTRAST_PAIRS`).
  - Styled from `tokens` (heading row tint, border, radius, `column_spacing`) so it matches the DS-1 card surfaces; wrapped by the view in a horizontally-scrollable container (a wide table must scroll inside its own region — mirrors the responsive-table discipline; a `ft.Row(scroll=...)` or `ft.Container` wrapper).
- Kept minimal (no sort/select/checkbox — `show_checkbox_column=False`, no `on_sort`): YAGNI for a read-only 2–3×/yr surface.

  1. `_greeting_header(app_config)` — a branded hero (reuse the `home.py` pattern / `components.card(gradient=hero_gradient())`), "Run History" + a plain subtitle greeting `friendly_district_name`.

## Risks & mitigations
- **A malformed/partial record crashes the row build (the #1 read-only risk).** → `to_run_row` reads every field via `.get` + `_as_int` (the same total discipline `home_status`/`convert_result` use), so a missing key → a safe default, never a `KeyError`; a totality test parametrizes over records-missing-every-key asserting no raise. The view additionally wraps read/derive/render in `ErrorCard`. TESTED across every axis.
- **The raw `error`/path leaks into the table (the privacy risk the Streamlit page HAS).** → `RunRow` carries NO `error` field at all; `status_label` is a fixed category label from structured fields only; the raw-log expander + log-path caption are DROPPED. A TESTED assertion: a fake-path `error` appears in NO `RunRow`/banner string. This is the concrete privacy fix vs Streamlit.
- **Empty vs unavailable vs error conflated (a "no runs yet" admin sees a scary error).** → three distinct states: `[]` → calm "no runs yet" WARNING (never red); `None` → "history unavailable" WARNING; an unexpected exception → `ErrorCard`. Each is a separate banner/branch, each TESTED (the derivation for the first two; the view wrapper for the third).
- **The banner + a row label contradict (banner says healthy, a row says failed).** → both derive from the SAME `home_status` precedence vocabulary (banner classifies the latest record; `to_run_row` classifies each record with the same rule order); the banner is about the LATEST run (`records[0]`), the rows show ALL — a healthy banner with an older failed row in the list is correct + honest, not a contradiction. Pinned in the Spec + a test asserting the latest row's label agrees with the banner's verdict category.
- **`ft.DataTable` API drift / remembered 0.2x forms.** → the 1.0 API was introspected against the installed `flet==0.85.3` (`columns`/`rows`/`cells`/`content`/`label`/`numeric` confirmed present); the `run_table` factory is the single place it's used (a future drift is one edit). Built via `components.py` per the conventions doc.
- **Sync read blocks the UI.** → rejected as a real risk: a kB local log parsed to a `list[dict]` is not a freeze (the same `read_run_records()` call `home.py` already makes on mount); the worker-thread convention is `run_pipeline`-scoped; async here is YAGNI. Documented (matches IA-3's decision).
- **A wide table forces horizontal page scroll.** → the `ft.DataTable` is wrapped in its own horizontally-scrollable region (the table scrolls inside its container, the page body never scrolls horizontally); the conditional myBlueprint+ columns keep a SpacesEDU district to 5 count columns (no needless width).
- **Coverage.** → `run_history.py` (the pure derivation) COUNTED (carries the 80% gate — a small pure module fully exercised); only `screens/run_history.py` added to the omit list (`components.py` already omitted). Matches the established `home_status`/`convert_result` COUNTED + `screens/*` omitted tiering.

## Design questions I had to decide (surfaced for the gate)
1. **Drop the raw `Error` column + log-path caption + raw-log expander (vs port them like Streamlit)?** → **DROP all three.** *Decided: privacy + product-ux.* The raw `error` is `str(e)` (path/`sis_type`/column risk); a non-technical admin doesn't read a stack trace; the category-only `status_label` (from structured fields) tells them what they need. This is the concrete fix for the Streamlit page's leak. *(A "for support" raw-log affordance is a ROADMAP item if ever field-justified — not this slice.)*
2. **Banner-first, or table-first (as Streamlit)?** → **Banner-first.** *Decided: product-ux / verdict-first.* Run History reinforces the same trust Home gives; the "is my sync OK?" verdict + staleness answer must precede the log (Streamlit led with the table and never showed a verdict).
3. **A new verdict vocabulary for the banner, or reuse `home_status`'s?** → **Reuse `home_status`'s precedence + voice.** *Decided: DRY + consistency.* The banner answers the same question over the same latest record; reusing the ordered rules (failed-ETL → failed-SFTP → anomaly → data-errors → stale → healthy) + the phrasing keeps Home + Run History from drifting. The one Run-History-specific bit is the per-run `RunRow` label (a table needs a per-row status, which Home never does).
4. **Reuse a `home_status` internal for the banner classification, or restate the 6 rules?** → **Deferred to Spec (a small choice).** The banner classifies the latest record with the same precedence `derive_home_status` uses; if a clean shared internal (e.g. a `_classify_latest(record) -> Verdict` extracted from `home_status`) can be reused without coupling Run History to Home's `HomeStatus`/`FixAction`/`HomeMetrics` types, prefer that (pure DRY); otherwise restate the 6 rules in `run_history.py` (they're short + the vocabulary is shared). Recommend: restate in `run_history.py` (the banner's headlines/details differ from Home's — "your LAST sync" framing vs Home's "your roster" — so a shared classifier would only save the verdict selection, not the copy; a small additive `home_status` refactor is acceptable if it's clean + tested, per the reuse guardrail). Pin at Spec.
5. **`ft.DataTable` sort/pagination?** → **Neither.** *Decided: YAGNI.* A read-only 2–3×/yr surface reviewing a short newest-first list (capped 50) needs no sort/filter/pagination; they'd add compute + surface for no proven need. Pagination is a ROADMAP note IF the log could grow large.
6. **Thread `on_navigate` into `build_run_history`?** → **No (minimal signature).** *Decided: KISS.* Run History is a read-only terminal surface with no fix-path CTA (unlike Home's "Open Setup"/"Check Run History"); `build_run_history(page, *, app_config)` is enough. Add `on_navigate` only if a future consumer needs it (build-at-first-consumer).
