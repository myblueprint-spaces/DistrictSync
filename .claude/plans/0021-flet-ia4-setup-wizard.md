## Problem
Setup is the security-sensitive heart of DistrictSync. The **folders step** landed (PLAT-2 — `screens/setup.py`: two `PickerField`s + a SIS/district dropdown → validate → gated Save → persist `input_dir`/`output_dir`/`sis_type` to `AppConfig`), which flips `is_complete()`. But an install is only *usable unattended* once it also **registers a Windows scheduled task** (the invisible nightly CLI — needs the account **password** to run logged-off) and, optionally, **stores SFTP credentials in the OS keyring** (so the nightly run can deliver to SpacesEDU). Today the Flet Setup surface stops at folders; the scheduler/SFTP/keyring/elevation machinery lives only in the Streamlit `01_Setup_Wizard.py` (deleted at CUT-1). Until IA-4, a Flet-only admin can pick folders but **cannot schedule the sync or set up delivery** — `nav.needs_setup(...)` never clears, and Home never leaves onboarding.

This is the **riskiest slice in the rebuild** because it handles two secrets on a live security boundary:
- The **Windows account password** — collected once to register the task for unattended (logged-off) operation. It must NEVER be logged, persisted to `AppConfig`/disk, put on a subprocess argv, or echoed in any success/error message. `register_task` already enforces this (child env `DSYNC_TASK_PW`, never argv; `_clean_ps_stderr` de-CLIXML + secret-strip). IA-4 is **presentation only** — it passes the password straight through and relies on the tested core's guarantees; it must not re-implement, weaken, or route around them.
- The **SFTP password** — stored in the OS **keyring** via `SFTPUploader` (the exact path the `--sftp-configure` CLI uses). Never in `AppConfig`, never logged.

Separately, `_classify_schedule_error(msg, elevated)` — the one piece of real, testable logic in the wizard (maps a registration failure + elevation state into a plain-language "run-as-admin vs batch-logon-right/wrong-password vs PowerShell-missing" message) — is **UI-layer code trapped in the Streamlit page** (`01_Setup_Wizard.py:76`), unreusable and tested only by loading that page by file path. It is presentation-neutral (no Streamlit), so IA-4 **relocates it to a pure Flet-neutral module** as its single source, with the Flet schedule step as its first Flet consumer (CUT-1's migrate-before-delete checklist names it).

### B. The scheduler section — a new sub-view in `screens/setup.py`
- **Windows path.** A `components.card` section: a heading, an `ft.TextField` for the run time (default from `AppConfig.schedule_time` = `"03:00"`; validated via `validators.validate_run_time` before Save — a pure boundary check, mirroring the folders-step `setup_state` gate discipline), the **run-as account display** (`current_run_as_user()`, a calm caption "This task will run as: **DOMAIN\user**"), and the **password field**: `ft.TextField(password=True, can_reveal_password=True, ...)` with a plain helper ("Lets the nightly sync run after a reboot with no one logged in. Used once to register the task — DistrictSync does not store it."). A persistent calm note states the blank-password consequence (logged-on-only).
- **On "Register schedule"** (a `components.primary_button`, gated `disabled` until folders are saved + `is_complete()` and the run-time field is non-empty — an invalid/blank time can't reach `register_task`):
  1. `run_time = field.value` → `validators.validate_run_time(run_time)` (raises `ValueError` → show a calm inline message, do not call the core). Persist `cfg.schedule_time = run_time` first (non-sensitive) so a partial state survives.
  2. Resolve `exe_path = Path(sys.executable)` (running python / frozen exe — same as Streamlit `_register_schedule`).
  3. `sys.platform == "win32"` → `ok, msg = register_task(task_name=cfg.schedule_task_name, exe_path=exe_path, sis_type=cfg.sis_type, input_dir=Path(cfg.input_dir), output_dir=Path(cfg.output_dir), run_time=cfg.schedule_time, sftp=cfg.sftp_enabled, run_as_user=None, run_as_password=(password or None))`. **The password flows UI field → `run_as_password` argument → (core) child env `DSYNC_TASK_PW`.** It is a local variable, never assigned to `cfg`, never logged, never in the message. `run_as_user=None` lets the core default to `current_run_as_user()` (matching Streamlit).
  4. `if ok:` set `cfg.schedule_registered = True`; `cfg.save()`; render a **`HealthVerdictBanner(Verdict.HEALTHY, ...)`** success ("Nightly sync scheduled — runs as **{account}**, whether or not you're logged in, daily at {time}."). **If `password` was blank** → a `Verdict.WARNING` banner ("Scheduled — but it will only run while you're logged in. Re-register with your Windows password for unattended operation across reboots.") — the Streamlit `st.warning` equivalent.
  5. `else:` compute `elevated = is_elevated()` (win32) and render `classify_schedule_error(msg, elevated)` in a **`components.ErrorCard`** (or `HealthVerdictBanner(Verdict.FAILED, ...)`) — the actionable "run as administrator / batch-logon-right / PowerShell-missing" message. `schedule_registered` stays False; the admin can fix + retry. The raw `msg` is only shown via the classifier's sanitized output.
- **Linux path (minimal, noted — not gold-plated).** `sys.platform != "win32"` → hide the password field + run-as caption entirely (cron has no per-task password), call `register_cron(exe_path, sis_type, input_dir, output_dir, run_time, sftp=cfg.sftp_enabled)`; on `ok` set `schedule_registered=True` + save; on failure show a plain "Couldn't create the schedule: {msg}" (no elevation classifier — the classifier is Windows-shaped, and `register_cron`'s failures are crontab errors). This matches the Streamlit `_register_schedule` else-branch. Windows is the primary target (districts run Windows servers); this keeps the Linux desktop path working without a bespoke Linux error taxonomy.

### C. The SFTP section — a new sub-view in `screens/setup.py`
- A `components.card` section with a toggle-like affordance (or always-visible fields with an "Enable SFTP delivery" primary action — decide at Spec; the simplest is always-visible fields + an explicit "Save SFTP credentials" button, since a 2–3×/yr admin sets it once). Fields: **host** (`ft.Dropdown` sourced from `sorted(ALLOWED_SFTP_HOSTS)` — the allowlist is the ONLY selectable set, so an out-of-allowlist host is structurally impossible), **username** (`ft.TextField`), **remote path** (`ft.TextField`, default `"/files"`), **port** (`ft.TextField`/number, default 22), **password** (`ft.TextField(password=True, can_reveal_password=True)`, placeholder "Leave blank to keep the existing stored credential").
- **On "Save SFTP credentials"** (`components.primary_button`, gated until host+username+remote present, and — first-time — a password present, mirroring the Streamlit required-field check):
  1. Construct `uploader = SFTPUploader(host, int(port), username, remote_path)` — **`__init__` runs `validate_sftp_host` and raises `ValueError` on an out-of-allowlist host**; catch it → calm inline message (belt-and-suspenders with the dropdown allowlist).
  2. `if password:` `uploader.store_password(password)` → **keyring**. Then **verify the round-trip**: `read_back = uploader.get_stored_password()`; if falsy → a `Verdict.FAILED`/`ErrorCard` ("Couldn't read the credential back on this account — SFTP uploads may fail. Try again, or run the app as this account."). If truthy → a `Verdict.HEALTHY` banner ("SFTP credentials stored and readable by **{current_run_as_user()}** — the nightly task runs as this account."). The password is a local variable → keyring only; never assigned to `cfg`, never logged.
  3. Persist NON-sensitive settings: `cfg.sftp_enabled=True; cfg.sftp_host=host; cfg.sftp_port=int(port); cfg.sftp_username=username; cfg.sftp_remote_path=remote_path; cfg.save()`.
- **"Test connection"** (adopt — a `components.secondary_button`): `SFTPUploader(host, int(port), username, remote_path)`; if a password was just typed, `store_password` it first (so the test uses the fresh credential — matches Streamlit Step 4); `ok, msg = uploader.test_connection()` → a `Verdict.HEALTHY`/`FAILED` banner with the (already user-safe) `test_connection` message. *Decision:* adopt over YAGNI because it reuses one existing tested method, requires no new core, and materially de-risks the highest-consequence silent failure (a mistyped credential means the nightly roster never reaches SpacesEDU and the admin only learns via a broken Home verdict days later). `test_connection` opens a real (30s-timeout) network connection — run it off the UI thread per the `run_pipeline` worker-thread convention? **Decision at Spec:** `test_connection` is a blocking network call (up to 30s), so it MUST be marshalled via `page.run_thread` + `page.run_task` (the FLET conventions §"Worker-thread → UI marshalling") to avoid freezing the window — a small, bounded use of the already-documented pattern (NOT the deferred `JobRunner`; just the marshalling contract). This is the one place IA-4 touches concurrency; keep it minimal (a Test button spinner → result banner).

## The password security contract (first-class, TESTED)
These are the load-bearing invariants the security lens-review at Verify will scrutinize. Each is enumerated **and** named with how it is verified.

**Windows account password (schedule registration):**
- **I1 — flows only UI field → `register_task(run_as_password=...)` argument → (core) child env `DSYNC_TASK_PW`.** It is a **local variable** in `build_setup`'s handler; it is **NEVER** assigned to any `AppConfig` field, **NEVER** written to disk, **NEVER** logged, **NEVER** placed on a subprocess argv (the core passes it via child env, confirmed `windows.py:371-374`, `456-488`), and **NEVER** echoed in a success/warning/error message.
  - *Verified by:* a **manual "no password in AppConfig/logs" review** at Verify (grep the handler for any `cfg.<x> = password` or `logger.*password*`; confirm the only reference is the `register_task(run_as_password=password or None)` call site) + the architect core-untouched gate (the core's env-not-argv guarantee is unchanged) + the classifier leak-test (below).
- **I2 — the classifier never surfaces the password.** `classify_schedule_error(msg, elevated)`'s input is the **sanitized** `register_task` return (`_clean_ps_stderr` already de-CLIXML'd + dropped any `DSYNC_*`); the classifier only maps substrings + appends fixed copy, never the password.
  - *Verified by:* a **COUNTED leak-test** in `test_ui_flet_setup_errors.py` — pass a `msg` containing a fake secret token / path and assert it appears in the output ONLY where `msg` legitimately passes through (the `else` branch), and that the classifier adds no credential text; assert the known-substring branches (`PowerShell not found`, etc.) return fixed copy independent of `msg`. IA-4 does NOT re-test the core's sanitization (that's `windows.py`'s tested contract) — it tests that the classifier is a faithful, non-leaking mapper.
- **I3 — the password is not retained after registration.** No session-state persistence of the password beyond the handler's local scope (contrast the Streamlit page, which parked it in `st.session_state` across steps — the single-scroll Flet form has no cross-step gap, so the password is read from the field at Register-time and used immediately, never stashed).
  - *Verified by:* the same manual review (no module-level/closure variable holds the password after the handler returns).

**SFTP password (keyring):**
- **I4 — flows only UI field → `SFTPUploader.store_password(password)` → OS keyring.** Never assigned to `AppConfig`, never logged, never on argv. Only the NON-sensitive settings (`sftp_host`/`port`/`username`/`remote_path`/`sftp_enabled`) reach `cfg.save()`. This mirrors `main._sftp_configure` exactly (single behaviour across CLI + UI).

## Design questions I had to decide (surfaced for the gate)
1. **Classifier's new home + name** → `src/ui_flet/setup_errors.py`, `classify_schedule_error` (drop the leading underscore — it's now a public, reusable pure fn). *Decided: a neutral `ui_flet` module* (matches `humanize`/`nav`/`verdict`/`run_log` — UI-read presentation logic, not core). The Streamlit page keeps a **shim** (`_classify_schedule_error = classify_schedule_error`, or a delegating wrapper) so its file-path test stays green — realizing "the Streamlit page imports from the new home."
2. **Composition: sectioned single scroll vs multi-page stepper.** *Decided: sectioned single scroll* (folders → schedule → SFTP, each a `components.card`). *Rejected: a multi-step wizard-within-Setup* — the Streamlit 5-step flow existed because Streamlit reruns the whole script per interaction and needed `session_state` to persist across steps; Flet holds state in the live view, so a single scroll is simpler, has no cross-step password-parking gap (a security win — see I3), and suits a 2–3×/yr admin who sees the whole setup at once. A stepper would add navigation surface + a place to stash the password for no benefit (YAGNI).
3. **"Test connection" — adopt or YAGNI.** *Decided: adopt.* It reuses one existing tested method (`SFTPUploader.test_connection`), needs no core change, and de-risks the highest-consequence silent failure (a wrong credential → the nightly roster never reaches SpacesEDU). Cost is bounded: one button + the already-documented worker-thread marshalling for its 30s network call.
4. **Where the marshalling lives.** *Decided: inline in the SFTP section's Test handler via `page.run_thread`/`page.run_task`* (the documented contract, minimal use). NOT the deferred `JobRunner` (that's IA-5's `run_pipeline` surface) — a single bounded network call doesn't warrant the runner abstraction.
5. **Management (edit/disable schedule, edit SFTP) view.** *Decided: out of scope* — IA-4 ports first-time setup; the Flet surface is idempotent (re-open → re-register), covering the real need. A dedicated manage view is a ROADMAP item if field signal demands it.
