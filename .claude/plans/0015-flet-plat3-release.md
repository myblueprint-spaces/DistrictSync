## Problem
`.github/workflows/release.yml` still builds the **old Streamlit** exe: a **console** (`--onefile`, no `--noconsole`) `pyinstaller src/main.py` with `--collect-all streamlit`, `--add-data src/ui`, tkinter, and the full `src.*` hidden-import set, on three OS jobs pinned to **Python 3.11** (stale — the project is `requires-python = ">=3.13"`). PLAT-0/0b proved `flet pack` produces a **windowed, no-console, offline-client-embedded** artifact on Windows + macOS + Linux on real CI — but that proof ran against the **throwaway prototype** (`app.py`) on a **fork branch**, was never folded into the shipping pipeline, and the real Flet shell (`src/ui_flet/launcher.py`, landed in PLAT-1) has **no** packaged build or CI smoke. Three pieces the spec language assumed were reusable were in fact **never landed** (recon-confirmed): the WM_CLOSE+psutil zero-orphan close harness (prose only — run live in PLAT-0, never committed; the exact close API + timeout were never recorded), the real-launcher windowed smoke, and `ci_verify_pack.py` (fork-branch-only, against the prototype). PLAT-3 productionizes all of it.

## Approach
**Two release artifacts during the PLAT-3→CUT-1 window** (decision 1):
- **Stable:** the existing 3 Streamlit console jobs (`build-windows/linux/macos`) stay as-is **except** a Python `3.11 → 3.13` bump (align with `requires-python` + `ci.yml` + the PLAT-0b matrix). Same `pyinstaller src/main.py`, same artifacts (`DistrictSync-windows.exe`, …). This is the rollback floor and the nightly CLI/scheduled-task exe.
- **Preview:** a new windowed Flet exe built by **`flet pack src/ui_flet/launcher.py --name DistrictSync-flet`**. **Why `launcher.py`, not `main.py`:** `launcher.py` IS the real Flet UI entry (no `main.py` dual-mode dispatch), so (a) it sidesteps the Streamlit-default conflict entirely — a `main.py` exe with Streamlit stripped would crash on the no-argv default path (`_select_ui_launcher("")` → `import src.ui.launcher` → `ImportError`), and (b) it is exactly the target the PLAT-1 spec named for this smoke, and (c) `main.py` is never touched, so the core-untouched gate is trivially met. CUT-1 (not PLAT-3) repackages `main.py` as the windowed Flet-default dual-mode exe once Streamlit is removed.

**One reusable matrix workflow** (decision 3): `.github/workflows/flet-pack.yml` (`on: workflow_call`) runs the per-OS pack+smoke:
- `windows-latest` · `ubuntu-22.04` (pinned — the Linux client filename encodes the distro; `ubuntu-latest` would drift the cache key) · `macos-latest`; Python **3.13**.
- Linux only: `apt-get install -y libsecret-1-0 libgtk-3-0 libgl1 libglib2.0-0 xvfb` (the Flet desktop client links these; PLAT-0b finding) and run the smoke under `xvfb-run -a`.
- Install the **pinned** stack from `requirements.txt -r requirements-dev.txt` (single-source pins — `flet/flet-desktop` runtime + `flet-cli`/`flet-web` build-only; `test_flet_pin` already gates drift) **+** `pyinstaller pefile psutil` (smoke deps).
- `actions/cache` `~/.flet` keyed `flet-client-${{ runner.os }}-0.85.3-<flavor>` + a headless **`python -c "import flet_desktop; flet_desktop.ensure_client_cached()"`** warm-up before pack (PLAT-0 pre-seed; cache-miss self-downloads → fails loud, never a runtime-download exe). Linux flavor stays the default **`light`** (PLAT-0b proved it; no media needs `full`).
- `flet pack src/ui_flet/launcher.py --name DistrictSync-flet --paths . --add-data "config${SEP}config" --hidden-import …` with a **per-OS `--add-data` separator** (`;` Windows / `:` POSIX — a matrix var; the current release.yml already splits this) and the **full `src.*` + flet** hidden-import set: `flet`, `flet_desktop`, `src.ui_flet.{launcher,shell,nav,tokens,theme}`, `tkinter` (launcher's error-dialog fallback), the ETL set (`pandas`, `pydantic`/`pydantic_core`, `yaml`, `logging.config`, `src.etl.transformers.{registry,context,base,students,staff,family,classes,enrollments,blended,course_info,student_courses,student_attendance}`), `src.scheduler.{windows,linux}` (win) / `src.scheduler.linux` (lin+mac), `src.config.{app_config,loader}`, `src.utils.*`, the per-OS keyring backend (`keyring.backends.Windows` / `keyring.backends.macOS` / `keyring.backends.SecretService`+`libsecret`), **plus `--exclude-module streamlit --exclude-module src.ui`** (guarantee the Streamlit stack never bloats the Flet exe). Record the **real measured size** (PLAT-0's ~58 MB was flet-only; this adds pandas/pydantic/paramiko/keyring).
- Run `python scripts/ci_flet_pack_smoke.py <dist> DistrictSync-flet --require-close` on Windows (gating close) / without `--require-close` on Linux+macOS (report-only close); upload the artifact.

**One landed smoke script** `scripts/ci_flet_pack_smoke.py` — productionizes + extends the fork-branch `ci_verify_pack.py` against the real launcher, in a **single launch** that proves both offline-embed and clean close:
1. **[Windows]** `pefile` → assert `OPTIONAL_HEADER.Subsystem == 2` (WINDOWS_GUI / no console).
2. Move `~/.flet` aside + set `FLET_CLIENT_URL=http://127.0.0.1:9` (unreachable) → launch the artifact. Poll ≤45 s for `~/.flet/client` to reappear → **offline-embed PASS** (the only way the cache repopulates with download impossible is extracting the *embedded* bundle; this also proves the window booted). **Gating on all three OS.**
3. **Zero-orphan close:** collect the process tree (launched root + `psutil` recursive descendants), post a real **`WM_CLOSE`** to every visible top-level window owned by a tree PID (Windows, pure `ctypes` `user32` `EnumWindows`+`GetWindowThreadProcessId`+`PostMessageW` — **no pywin32 dep**), wait ≤ N s for the whole tree to exit, then re-sweep for **stray `flet.exe`**. Assert zero orphans. **`--require-close` → gating (Windows); absent → report-only (Linux/macOS):** on non-Windows the harness has no portable `WM_CLOSE`, so it does a clearly-labeled best-effort `terminate()`-based teardown check and **always exits 0**, printing the verdict (closes the conventions-doc open item where feasible without flaky release blocks).
4. Restore `~/.flet` in `finally`.

**Anti-rot + verify trigger** (decision 3): `release.yml`'s new `build-flet` job + a new `flet-verify.yml` (`on: workflow_dispatch` + `pull_request` paths: `src/ui_flet/**`, `src/main.py`, `requirements*.txt`, `pyproject.toml`, `.github/workflows/flet-pack.yml`, `scripts/ci_flet_pack_smoke.py`) BOTH `uses: ./.github/workflows/flet-pack.yml`. The verify exercises the build on every relevant PR (not just tags); `publish-release` attaches the Flet artifact as a **clearly-labeled preview** alongside the stable Streamlit downloads.

### Required changes (fold into the Spec before approval)
1. **[R1 — blocker] Build-time embed PROOF, not just the runtime smoke.** A fresh `flet_desktop@0.85.3` ships **no** bundled archive (`flet_desktop/app/` is absent); `flet pack` only embeds the client by copying a **populated** `~/.flet/client/…` into the bundle. So: (a) after the warm-up, assert `~/.flet/client/flet-desktop-<flavor>-0.85.3` exists + non-empty → **fail the job** if not (the warm-up's download fallback succeeds silently, so a miss is invisible without this); (b) after `flet pack`, assert the packed artifact actually contains `flet_desktop/app/<artifact>` (inspect the PyInstaller build manifest / onefile) → **fail** if absent. The runtime move-aside smoke is necessary-but-not-sufficient.
2. **[R2 — blocker] Harden the `~/.flet` restore-on-failure.** Tear down the **full process tree BEFORE** restoring (a Flutter-view subprocess mid-extraction holds `~/.flet` handles → the spike's `rmtree`+`move` in `finally` can throw and leave `~/.flet` destroyed). Restore defensively + idempotently (`os.replace` into a fresh path, try/except); on restore failure **fail loud** with "runner `~/.flet` may be corrupted" — never swallow. State the GitHub-hosted **ephemeral-runner** assumption + harden so it's safe if ever on a reused/self-hosted runner (`workflow_dispatch`/fork paths).
3. **[R4 — blocker] Fix the (never-landed) close harness against the real `flet pack` 3-process model** (`exe → re-exec'd Python host → separate `flet`/`flet.exe` view with its own PID file`): (a) **wait** (bounded) for a top-level window owned by a tree PID to **exist** before posting `WM_CLOSE` — "window never painted" is its own distinct failure, not a close failure; (b) primary orphan target = the **view's PID file** (deterministic), with (c) a **baseline-delta** name-sweep (snapshot `flet`/`flet.exe` PIDs BEFORE launch; only act on *new* ones — safe on shared runners) as backstop; (d) walk descendants of **both** the launched PID and any re-exec'd host child (by image name `DistrictSync-flet`).
