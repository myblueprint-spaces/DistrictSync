## Problem
Convert is the admin's **manual "run it now"** surface: pick the input GDE files → run the ETL → read a verdict + counts → optionally deliver via SFTP. The nightly scheduled CLI is the real engine; Convert is the on-demand escape hatch (a mid-day re-run after a data fix, a first proof the setup works). Today the Flet `convert` destination is a **placeholder** (`build_placeholder` in `shell.py`); the ad-hoc flow lives only in Streamlit `02_Convert.py` (deleted at CUT-1).

This is the **second-riskiest slice in the rebuild** (after IA-4) because it is the FIRST surface to run the **blocking pandas ETL on a WORKER THREAD**. The `JobRunner` abstraction was DEFERRED to IA-5 as its first real `run_pipeline`-shaped consumer (the conventions doc's "promote on 2nd use" — IA-4b proved the raw `page.run_thread`/`page.run_task` marshalling for a `(bool,str)` network call; IA-5 generalizes it for a call that can `SystemExit`). The #1 correctness trap is REAL here:
- **`run_transform`/`run_pipeline` block** (pandas) → they MUST run off the UI thread via `page.run_thread`, or the window freezes for the whole conversion. UI updates MUST marshal back via `page.run_task` — **never mutate a Flet control from the worker thread** (corrupts/freezes the window).
- **The `SystemExit`-vs-`Exception` asymmetry:** if IA-5 calls `run_pipeline`, bad input triggers `sys.exit(1)` (`pipeline.py:294/302/305`). `SystemExit` is **NOT** an `Exception` subclass — a bare `except Exception` in the worker lets it **propagate and silently kill the worker thread**; and on that path `run_pipeline` re-raises the `SystemExit` at `:420` **before** `_emit_run_log`, so **NO `__DISTRICTSYNC_RUN__` record is written** → the UI's `on_error` is the **ONLY** failure signal. Get this wrong and a bad-input run vanishes with no verdict.
- **The exit-3 SFTP shape:** a *successful ETL with a failed SFTP upload* is **not an exception** — `run_pipeline` returns a `PipelineResult(sftp_attempted=True, sftp_ok=False)`. The UI must read those booleans in `on_done` and render an honest "built but not delivered" verdict, NOT treat it as `on_error`.

Get the worker→UI marshalling + the `SystemExit`/`Exception` asymmetry + the exit-3 booleans wrong and the window freezes, corrupts, or silently swallows a failure. IA-5 must bake all three into a **tested `JobRunner` contract**.

## Goals / Non-goals
- **Goal — a reusable `JobRunner` (the worker-thread wrapper; its first real consumer).** Runs a blocking callable OFF the UI thread via `page.run_thread`, marshals `on_done(result)` / `on_error(exc)` back via `page.run_task`, and **correctly handles the `SystemExit`-vs-`Exception` asymmetry** (catches `SystemExit` SEPARATELY — the only failure signal on the bad-input path — and surfaces it to `on_error`; a caught `Exception` also surfaces to `on_error`). The **trust-critical state machine** (idle → running → done/error, single-flight) is extracted as a COUNTED pure helper; the flet `page.run_thread`/`page.run_task` wiring is the coverage-omitted glue.
- **Goal — the Convert view** (coverage-omitted): reuse `PickerField`/`pick_files` for input selection (with the landed boundary validation); a **FileChip** component (built here in `components.py`, first consumer) per selected file; a "Convert now" action gated on valid inputs; a running state (indeterminate spinner — the window must not freeze); a **human verdict** result (reuse the DS-1 `HealthVerdictBanner` + IA-3's verdict vocabulary) showing entity counts + any data-errors/anomalies in plain language.
- **Goal — SFTP pre-flight confirm:** if SFTP is configured (`AppConfig.sftp_is_configured()`), an EXPLICIT confirm before delivering (don't silently upload); surface the exit-3 shape (ETL ok but SFTP failed → a clear "built but not delivered" verdict, read from the result booleans, NOT an exception).
- **Goal — >20% anomaly acknowledgment:** if `compute_anomalies` reports a >20% drop, require the admin to **acknowledge** before the result is accepted/delivered — the anomaly is a data-safety guard; a silent 20% roster drop must not ship.
- **Goal — the write-in-flight close guard** (attach at the `_on_leave` seam PLAT-1/IA-2 reserved for IA-5): closing the window mid-write shouldn't corrupt output. The loader's backup-and-restore `save_all` atomicity is the real safety net (state that); the guard is a reassurance/confirm, NOT a re-architecture of the close path (keep the zero-orphan close intact).
- **Non-goal — ANY change to `src/etl|config|sftp|scheduler|main.py`.** `run_transform`/`run_pipeline`/`DataLoader`/`compute_anomalies`/`SFTPUploader`/`DataExtractor` are CALLED, never modified. Asserted as an **architect core-untouched gate** + `git diff --stat`. (A genuine core gap → a ROADMAP item, not folded here.)
- **Non-goal — rebuilding the picker.** `PickerField`/`pick_files`/`validate_input_dir` LANDED in PLAT-2 — REUSED verbatim.
- **Non-goal — the Streamlit diff/quality-report richness beyond a cheap reuse.** The diff (`_compute_diff`) + quality report (`DataQualityReport`) are ported ONLY if they reuse existing compute cheaply (decision below — the quality report is one existing call; the diff is deferred as a collapsible-nice-to-have, not core to the verdict). No progress-percentage (the ETL emits no progress → an indeterminate spinner is correct, not a fake bar).
- **Non-goal — a second SFTP credential surface.** IA-4b owns credential *setup* (keyring store + test). IA-5 only *uses* stored credentials to *deliver*; if `not sftp_is_configured()` the SFTP step is simply absent (matching Streamlit).

## The core-entry decision (load-bearing — decided + justified against the parity lock)

**DECISION: Convert calls the `run_transform` + `DataLoader.save_all` ADAPTER path (mirroring the Streamlit page), NOT `run_pipeline`.** The blocking work the `JobRunner` runs is a small orchestration function (call it `convert_job`, a COUNTED-boundary helper — see §A) that does exactly what the Streamlit `run_conversion` + the SFTP step do, in sequence, off the UI thread.

**Why the adapter, not `run_pipeline`:**
1. **Byte-parity is proven for the adapter, not for a fresh `run_pipeline` UI call.** `tests/test_pipeline_parity.py` locks `load_from_bytes → run_transform → DataLoader.save_all` byte-for-byte against `run_pipeline` (incl. the per-entity BOM rule). The Streamlit Convert deliberately took this path (per `docs/claugentic-DECISIONS.md` 2026-06-23, Plan 0005 Slice 2b) so the UI can never diverge from the CLI. Reusing it keeps IA-5 inside that lock; introducing `run_pipeline` as the UI's Convert engine would be a NEW output path the parity test doesn't cover.
2. **The adapter gives the view the frames it needs** — `TransformOutputs.outputs`/`field_orders`/`data_errors` — for the counts, the data-errors banner, the quality report, and `compute_anomalies` (which takes `outputs` + `output_dir`). `run_pipeline` returns only a `PipelineResult` (counts + booleans), hiding the frames the anomaly-ack + quality report need.
3. **Anomaly-ack requires computing anomalies BEFORE writing/uploading.** `compute_anomalies(outputs, output_dir)` is pure (compares `outputs` against the existing CSVs on disk). The adapter path lets IA-5 compute anomalies **after transform, before `save_all`/upload** — exactly where the ack gate belongs. `run_pipeline` computes + acts on anomalies internally with no ack hook.
4. **SFTP stays an explicit, ack-gated, pre-flight-confirmed second step** (as in Streamlit) rather than a `run_pipeline(sftp=True)` side effect — this is what makes the pre-flight-confirm + exit-3 honesty possible in the UI.

**The `SystemExit` asymmetry still applies — deliberately, as a SAFETY NET, not because the adapter calls `sys.exit`.** `run_transform`/`load_from_bytes`/`save_all` raise ordinary `Exception`s (e.g. `save_all`'s `ValueError` on a missing field-map column, `load_config`'s `FileNotFoundError`/`ValueError`, `SFTPUploader`'s `RuntimeError`) — they do NOT call `sys.exit`. **BUT** the `JobRunner` MUST STILL catch `SystemExit` separately, because (a) `load_config` is called inside `convert_job` and a future core change could route a `sys.exit` through a shared helper, and (b) the conventions doc pins the `SystemExit` catch as the *contract* for any worker running ETL-shaped code — the `JobRunner` is reusable and its NEXT consumer may call `run_pipeline` directly. **So: the `JobRunner` catches `SystemExit` → `on_error` unconditionally (defense-in-depth + contract), even though the adapter path is not expected to raise it.** The exit-3 shape is handled by the adapter returning a structured result the view reads (see §A `ConvertResult`), never via `on_error`.

*(Rejected alternative — `run_pipeline`: it would be the "full nightly-equivalent" incl. SFTP + run-log + internal anomaly handling, but it (a) is outside the parity lock as a UI path, (b) hides the frames the view needs, (c) offers no ack hook for the >20% guard, and (d) makes the exit-3 pre-flight-confirm impossible. It IS the right engine for the nightly CLI — which is exactly why the UI must not re-run it as its manual path. Noted, rejected, justified.)*

## Approach

### A. The `JobRunner` + its pure state machine + the `convert_job` boundary
- **`src/ui_flet/job_runner.py` — the state machine (COUNTED) + the runner glue (coverage-omitted, in the SAME file with a `# pragma: no cover` on the flet-touching method).** Mirrors `filepicker.py`'s split (pure validation COUNTED, `await`-dialog glue `# pragma: no cover`):
  - **`JobState` (COUNTED, pure `Enum`):** `IDLE`, `RUNNING`, `DONE`, `ERROR`.
  - **`JobStateMachine` (COUNTED, pure, no flet import):** the trust-critical single-flight logic — `start()` (IDLE→RUNNING, returns `False` if already RUNNING so a double-click can't launch two conversions), `finish_ok()` (RUNNING→DONE), `finish_error()` (RUNNING→ERROR), `reset()` (DONE/ERROR→IDLE for a re-run), plus `is_running` / `can_start` properties. Total + tested: illegal transitions (e.g. `finish_ok()` from IDLE) return a bool/no-op rather than corrupting state. **This is the guard against the double-click-launches-two-ETLs race** — the single trust-critical concurrency invariant that is cheaply testable headless.
  - **`JobRunner` (glue, `# pragma: no cover` on the run method):** `run(page, work, *, on_done, on_error)` — asserts `state.start()` (no-op if already running), then:
    ```python
    def _work() -> None:                          # runs OFF the UI thread
        try:
            result = work()                       # blocking (convert_job / run_pipeline)
        except SystemExit as ex:                  # NOT an Exception subclass — catch FIRST
            page.run_task(_deliver_error, ex)     # the ONLY signal on the sys.exit(1) path
            return
        except Exception as ex:                   # noqa: BLE001 — surface every failure to the UI
            page.run_task(_deliver_error, ex)
            return
        page.run_task(_deliver_done, result)      # marshal success back to the loop
    async def _deliver_done(result): state.finish_ok(); on_done(result)
    async def _deliver_error(exc):   state.finish_error(); on_error(exc)
    page.run_thread(_work)
    ```
    UI mutation happens ONLY inside `on_done`/`on_error` (called from the `page.run_task` coroutines the loop owns) — **never from `_work`**. The `SystemExit` catch is FIRST and separate (it would otherwise slip past `except Exception` and kill the thread silently).
- **`convert_job` — the boundary orchestration the `JobRunner` runs (in `screens/convert.py`, view-glue, but a thin, single-purpose function; its logic mirrors the parity-locked Streamlit `run_conversion` + SFTP step).** Given the selected input dir/files + config name + an `sftp_requested` flag + an `anomaly_ack` flag, it (off the UI thread):
  1. `load_config(config_name)` → `to_raw_dict()` → `mappings`/`global_config`/`file_headers` (collect headerless headers exactly as Streamlit + the parity test do).
  2. Read the selected files' bytes → `DataExtractor("").load_from_bytes(sources, file_headers)` (encoding/delimiter core). Empty/unloadable → return a `ConvertResult(status=NO_INPUT, …)` (a structured result, NOT a raise — mirrors Streamlit's "No files could be loaded").
  3. `outputs, field_orders, data_errors = run_transform(raw_data, mappings, global_config)`. Empty outputs → `ConvertResult(status=NO_OUTPUT, …)`.
  4. `anomalies = compute_anomalies(outputs, Path(output_dir))` (against the existing output CSVs). **If `anomalies and not anomaly_ack` → return `ConvertResult(status=NEEDS_ANOMALY_ACK, anomalies=…, outputs=…)` WITHOUT writing** — the view then shows the ack gate and re-invokes with `anomaly_ack=True`.
  5. `DataLoader(output_dir).save_all(outputs, field_orders)` — the atomic write (raises `ValueError` on a missing field-map column → propagates to `on_error`).
  6. Quality report: `DataQualityReport().analyze(outputs).to_text()` (one existing call — cheap; carried into the result for a collapsible).
  7. If `sftp_requested`: `SFTPUploader(host, port, username, remote_path).upload_csvs(Path(output_dir), sis_type=config_name)`; a `RuntimeError` here is caught INSIDE `convert_job` and folded into `ConvertResult(sftp_attempted=True, sftp_ok=False, sftp_error=<category>)` — **the exit-3 shape as a RESULT, not an exception** (a failed delivery must not discard the successful ETL; the files are already written). ETL-level failures (steps 1–5) DO propagate as exceptions → `on_error`.
  - Returns a **`ConvertResult`** dataclass (see §B) the view renders verdict-first.
- **Why the state machine is COUNTED and the runner is omitted:** the state machine is pure frozen/enum logic (the double-launch guard is the trust-critical bit, headless-testable); the `page.run_thread`/`page.run_task` wiring can only be exercised with a live loop (matches the `filepicker` await-glue omit precedent).

### B. `ConvertResult` + verdict mapping (COUNTED pure summary)
- **`src/ui_flet/convert_result.py` — a pure COUNTED module (no flet import), mirroring `home_status.py`.** Defines:
  - `ConvertStatus` (Enum): `DELIVERED` (ETL ok + SFTP ok or not requested), `BUILT_NOT_DELIVERED` (ETL ok, SFTP attempted + failed — the exit-3 shape), `NEEDS_ANOMALY_ACK`, `NO_INPUT`, `NO_OUTPUT`, `BUILT_WITH_DATA_ERRORS` (ETL ok, data-errors present).
  - `ConvertResult` (frozen dataclass): `status`, `entity_counts: dict[str,int]`, `data_errors: list[dict]`, `anomalies: list[str]`, `sftp_attempted: bool`, `sftp_ok: bool`, `quality_text: str` (+ the transient `outputs`/`field_orders` the view needs for a re-invoke on ack — kept off the frozen summary or carried as opaque; decide at Spec to keep the pure summary flet-free and PII-free).
  - **`summarize(result) -> (Verdict, headline, detail)` — the pure, TOTAL, TESTED mapping** from a `ConvertResult` to the DS-1 verdict vocabulary (reusing `Verdict` + the plain-language voice `home_status` established): `DELIVERED` → `Verdict.HEALTHY` ("Roster converted and delivered to SpacesEDU" or "Roster converted" when SFTP not requested); `BUILT_NOT_DELIVERED` → `Verdict.FAILED` ("Your roster was built but didn't reach SpacesEDU" — mirrors `home_status`'s exit-3 headline); `BUILT_WITH_DATA_ERRORS` → `Verdict.WARNING` ("Converted with N data warnings"); `NEEDS_ANOMALY_ACK` → `Verdict.WARNING` ("Some files look much smaller than usual"); `NO_INPUT`/`NO_OUTPUT` → `Verdict.FAILED` ("No files could be read"/"No output produced — check the district"). **Privacy: NEVER interpolate a raw file path / `sis_type` / column name into a headline/detail** (same rule `home_status` enforces) — faults are named by category; the raw error belongs to the log only.
- **DECISION — reuse IA-3's verdict vocabulary, don't invent a new one.** The Convert result and the Home health verdict answer the same admin question in the same voice; reusing `Verdict` + `HealthVerdictBanner` + `home_status`'s phrasing (esp. the exit-3 "built but not delivered" headline) keeps setup/health/convert feedback consistent. The one Convert-specific addition is the `NEEDS_ANOMALY_ACK` transient state (Home never needs an ack). So: reuse the spine, add only the ack status. *(Rejected: a bespoke `convert_verdict` vocabulary — it would drift from Home for no benefit.)*

### C. The Convert view — `src/ui_flet/screens/convert.py` (coverage-omitted)
- **Layout (a sectioned single scroll, DS-1 spine, mirroring `setup.py`'s composition):**
  1. A branded hero header (`components.card(gradient=hero_gradient())`) — "Convert now" + a plain subtitle.
  2. **District display/select** — sourced from `available_configs()` with `friendly_district_name` (defaults to `AppConfig.sis_type`); a read-only caption if already configured, or a small dropdown (KISS — reuse the `setup.py` `_district_options` pattern; do NOT duplicate — extract a shared `_district_options` if it's used twice, else inline).
  3. **Input selection** — a `PickerField` (REUSED) picking the input FOLDER (default `AppConfig.input_dir`), validated by `validate_input_dir`; PLUS the picked files shown as **FileChips**. *Design note (below):* the Streamlit page uploads individual files; the Flet folder-based flow picks a folder and lists the GDE files found in it (matching the CLI + the nightly run, which read a folder). The FileChips list the resolved GDE files (from `extract_required_files` expectation + what's present); a **missing-file warning** (from `extract_required_files`) shows any expected-but-absent file plainly.
  4. **"Convert now"** — `components.primary_button`, gated `disabled` until the input dir validates (structural gate, mirroring `setup.py`'s Save gate) AND the state machine `can_start` (no in-flight run).
  5. **A running state** — on click: `state.start()`, disable the button, show an indeterminate `ft.ProgressRing` + a calm "Converting… this can take a moment for large extracts" caption; `JobRunner.run(page, work=lambda: convert_job(...), on_done=…, on_error=…)`.
  6. **A result slot** — swapped on `on_done`/`on_error` to a `HealthVerdictBanner` (from `summarize`) + entity-count tiles (reuse `home.py`'s `_metric_tile` pattern — extract a shared `_metric_tile`/`_metric_tiles_row` into `components.py` if it's now used by both Home and Convert; DRY) + a collapsible quality report + the data-errors line (plain language, from `data_errors`).
- **The anomaly-ack gate (a real safety gate, not a nuisance):** when `on_done` receives `status=NEEDS_ANOMALY_ACK`, render a `Verdict.WARNING` banner listing the anomalies in plain language (reuse `home_status._anomaly_detail`-style phrasing — NOT the raw `ANOMALY:` string) + an explicit **"I've reviewed this — convert anyway"** `secondary_button` and a **"Cancel"**. The ack button re-invokes the SAME `JobRunner.run` with `convert_job(..., anomaly_ack=True)` (which then proceeds past step 4 to write). No silent 20% drop can ship. (The write has NOT happened yet at the ack point — `convert_job` returns before `save_all` when `NEEDS_ANOMALY_ACK`.)
- **The SFTP pre-flight confirm:** the SFTP step is shown ONLY if `AppConfig.sftp_is_configured()`. It is a SEPARATE explicit action *after* a successful build (mirroring Streamlit's separate "Upload via SFTP" button), NOT auto-run. On click, show a **confirm** (`page.show_dialog` with an `ft.AlertDialog`, per the conventions doc) naming the destination (`host:port/remote_path` — a config value, never a secret) — "Deliver the converted roster to `<host>`?" with Deliver/Cancel. On Deliver, re-invoke `JobRunner.run` with `convert_job(..., sftp_requested=True)` (which re-transforms + writes + uploads in one job — or, to avoid re-transforming, decide at Spec whether `convert_job` can take an already-built `outputs` for the upload-only leg; simplest correct is a dedicated `upload_job` that writes-then-uploads from the retained frames — see §D). The exit-3 result (`sftp_attempted=True, sftp_ok=False`) renders the `BUILT_NOT_DELIVERED` FAILED verdict — read from booleans, never `on_error`.

### D. The write-in-flight close guard (at the `_on_leave` seam)
- **Attach at `shell._on_leave(page)`** (the reserved seam, `shell.py:118`) — a module-level "is a conversion writing right now?" flag the Convert view sets when `save_all` is about to run and clears when the job finishes. `_on_leave` reads it.
- **The UX (decided below):** the loader's backup-and-restore `save_all` atomicity is the REAL safety net — a mid-write close cannot tear the output (any incomplete commit rolls back). So the guard is a **brief reassurance, not a hard block**: if a write is in flight when the window closes, `_on_leave` logs/notes it and lets the atomic close proceed (the file finishes atomically or rolls back cleanly). **DECISION — reassurance-only, no confirm dialog + no block** (see the design-question section): a 2–3×/yr admin closing mid-convert should not be trapped, and the atomicity means they can't corrupt anything; a confirm dialog on close would be a nuisance for a non-existent risk. The guard's value is (a) making the invariant explicit + tested (the flag flips correctly around `save_all`) and (b) a documented seam a future field-justified confirm could mount into. **Do NOT re-architect the close path** — `_on_leave` stays a fast, exception-safe hook; the zero-orphan `page.window.destroy()` is untouched.
- **Why not a hard block / confirm:** blocking the close waiting for a pandas write risks the exact freeze/zombie the whole Flet migration deleted; a confirm adds friction against a risk the loader already eliminates. State the atomicity as the safety net explicitly (the plan brief's own framing).

### E. Wiring
- **`shell.py` (ONE-LINE addition, mirroring the `home`/`setup` swaps):** `screens["convert"] = functools.partial(build_convert, page)` — `convert` is already a `DESTINATIONS` entry; the swap replaces its placeholder. `build_convert(page)` owns NO nav/lifecycle beyond the `JobRunner` (which takes `page`); the write-guard flag lives in a small `shell`-or-`convert` module-level holder the `_on_leave` seam reads (decide the exact home at Spec — a `convert` module flag imported by `shell._on_leave`, keeping shell thin). **No other shell change.**

## The concurrency contract (first-class, enumerated + verified — the LIVE weight this slice)
Each invariant is named AND paired with how it is verified. This is the load-bearing surface the concurrency lens-review at Verify scrutinizes.

- **C1 — the blocking ETL runs OFF the UI thread.** `convert_job` (pandas `run_transform` + `save_all` + the SFTP network call) runs inside `page.run_thread(_work)`; the UI thread stays responsive (the window never freezes; the spinner animates).
  - *Verified by:* the `JobRunner.run` implementation (the ONLY call site of `work()` is inside `_work`, which `page.run_thread` schedules) + the manual smoke (the window is draggable/responsive during a real large-extract convert).
- **C2 — no Flet control is mutated from the worker thread.** Every UI update happens inside an `on_done`/`on_error` handler, which the `JobRunner` invokes ONLY from the `page.run_task` coroutines (`_deliver_done`/`_deliver_error`) the loop owns — `_work` calls `page.run_task(...)` and returns; it never touches a control.
  - *Verified by:* a code-review gate (grep `_work` for any control mutation / `page.update()` — must be zero; the only calls are `work()` + `page.run_task(...)`) + the IA-4b precedent (the same shape, reviewed).
- **C3 — the `SystemExit`-vs-`Exception` asymmetry is handled.** `_work` catches `SystemExit` in a SEPARATE `except` clause FIRST (it is not an `Exception` subclass; a bare `except Exception` would let it kill the thread silently) and routes it to `on_error`; a caught `Exception` also routes to `on_error`. On the bad-input `sys.exit(1)` path (if a future consumer calls `run_pipeline`), the UI's `on_error` is the ONLY failure signal — no run-log record exists (`pipeline.py:420` re-raises before `_emit_run_log`), so the view must not assume one.
  - *Verified by:* a **COUNTED test** in `test_ui_flet_job_runner.py` — a `work` that raises `SystemExit(1)` routes to `on_error` (not swallowed); a `work` that raises `Exception` routes to `on_error`; a `work` that returns routes to `on_done`. (The state machine + a fake/injected marshaller make this headless-testable — see Test strategy; the real `page.run_thread`/`page.run_task` is the omitted glue, but the catch-and-route logic is exercised via a thin seam.)
- **C4 — single-flight: a double-click cannot launch two conversions.** `JobStateMachine.start()` returns `False` (no-op) if already `RUNNING`; the button is also disabled on start. The state machine is the authoritative guard (the disabled-button is belt-and-suspenders).
  - *Verified by:* a COUNTED state-machine test (`start()` from `RUNNING` is a no-op returning `False`; only IDLE→RUNNING is allowed) + the button-disable in the view.
- **C5 — graceful degradation: a hung/failed run doesn't freeze the window.** A failure surfaces via `on_error` → a calm `ErrorCard`/`FAILED` banner + the button re-enabled (`state.reset()`); an SFTP failure is a `BUILT_NOT_DELIVERED` result, not a hang; the SFTP network call is bounded by `SFTPUploader`'s own timeout (as in IA-4b). The window is never blocked on the worker.
  - *Verified by:* the `on_error`/`on_done` re-enable path (state → IDLE, button re-enabled) + the manual smoke (a deliberately-broken config → a FAILED verdict, not a frozen window) + the write-guard (C6).
- **C6 — a window-close mid-write cannot corrupt output.** The write-in-flight flag + the loader's backup-and-restore `save_all` atomicity: an interrupted commit rolls back (the output dir is never torn); `_on_leave` reads the flag but does NOT block the atomic close.
  - *Verified by:* a COUNTED-adjacent assertion that the flag flips around `save_all` (the pure holder is testable) + the existing loader atomicity tests (unchanged core) + the manual smoke (close mid-convert → no torn output; re-open shows a clean state).

## Risks & mitigations
- **A `SystemExit` slips past `except Exception` and silently kills the worker (the #1 trap).** → The `JobRunner` catches `SystemExit` in a SEPARATE clause FIRST, routes to `on_error` (C3); a COUNTED routing test proves it; the conventions doc pins it. Even though the adapter path isn't expected to `sys.exit`, the catch is contract + defense-in-depth for a future `run_pipeline` consumer.
- **A UI control mutated from the worker thread freezes/corrupts the window.** → C2: the only worker-thread calls are `work()` + `page.run_task(...)`; all mutation is inside the marshalled coroutines; grep-gate + the IA-4b precedent.
- **The window freezes during a large convert.** → C1: the blocking work is on `page.run_thread`; an indeterminate spinner animates; the manual smoke drags the window mid-convert.
- **A double-click launches two conversions (a data race on `save_all`).** → C4: the state machine's `start()` is single-flight (no-op if RUNNING) + the button disables; COUNTED test.
- **A silent >20% roster drop ships to SpacesEDU.** → the anomaly-ack gate: `convert_job` returns `NEEDS_ANOMALY_ACK` WITHOUT writing when `compute_anomalies` fires + no ack; the view forces an explicit "reviewed — convert anyway" before re-invoking with `anomaly_ack=True`. A silent drop is structurally impossible.
- **A failed SFTP masquerades as success (or discards the built files).** → the exit-3 shape as a RESULT: `convert_job` catches the `SFTPUploader` `RuntimeError`, folds it into `ConvertResult(sftp_attempted=True, sftp_ok=False)`, and `summarize` maps it to a FAILED "built but not delivered" verdict — the files stay written (never rolled back for an upload failure); the admin can retry delivery.
- **A window-close mid-write corrupts output.** → C6: the loader's backup-and-restore `save_all` atomicity is the real net (rolls back a torn commit); the write-in-flight flag makes the invariant explicit; `_on_leave` does NOT block the atomic close (no freeze/zombie regression).
- **A missing field-map column writes a partial CSV.** → `save_all`/`select_ordered` raise `ValueError` (not `KeyError`) → propagates to `on_error` → a calm FAILED banner (fail-loud, matching the scheduled run + the parity test's `TestUIWriteFailsLoud`).
- **Accidentally changing the core.** → architect core-untouched gate + `git diff --stat`; the allowed file set is enumerated.
- **Coverage.** → `job_runner.py`'s state machine + `convert_result.py` COUNTED (carry the 80% gate — small pure modules fully exercised); the `JobRunner.run` method `# pragma: no cover` inline; `screens/convert.py` in the omit list. Matches the `filepicker` split + the view-omit pattern.

## Design questions I had to decide (surfaced for the gate)
1. **Core entry: `run_transform`-adapter vs `run_pipeline`.** *Decided: the `run_transform` + `DataLoader.save_all` ADAPTER* (mirroring Streamlit `run_conversion`). *Why:* it is inside the `test_pipeline_parity.py` byte-lock (a `run_pipeline` UI path is not); it exposes the frames the counts/anomaly-ack/quality-report need; it lets the anomaly-ack gate sit *between* transform and write; it keeps SFTP an explicit, pre-flight-confirmed, exit-3-honest second step. `run_pipeline` is the nightly engine — the UI must not re-run it as its manual path. Fully justified in §"The core-entry decision".
2. **The `JobRunner` split.** *Decided: a COUNTED pure `JobStateMachine` (the single-flight guard) + a `# pragma: no cover` `JobRunner.run` glue in the same file* (mirrors `filepicker.py`). The trust-critical bit (double-launch guard + transition legality) is headless-testable; the `page.run_thread`/`page.run_task` wiring is omitted. The `SystemExit`/`Exception` catch-and-route logic is exercised via an injectable-marshaller seam so the asymmetry is a COUNTED test, not just glue.
3. **Verdict vocabulary: reuse IA-3 or invent.** *Decided: reuse* `Verdict` + `HealthVerdictBanner` + `home_status`'s plain-language voice (esp. the exit-3 "built but not delivered" headline); add ONLY the Convert-specific `NEEDS_ANOMALY_ACK` status. Convert + Home answer the same question in the same voice — a bespoke vocabulary would drift for no benefit.
4. **The write-in-flight guard UX: block / confirm / reassure.** *Decided: reassurance-only — NO hard block, NO confirm dialog.* The loader's backup-and-restore `save_all` atomicity already makes a mid-write close safe (a torn commit rolls back); blocking risks the freeze/zombie the Flet migration deleted, and a confirm adds friction against a non-existent corruption risk. The guard's value is the explicit, tested invariant (the flag flips around `save_all`) + a documented seam a future field-justified confirm could mount into. `_on_leave` stays fast + exception-safe; the zero-orphan close is untouched.
5. **Input model: file-upload (Streamlit) vs folder-pick (Flet/CLI).** *Decided: folder-pick* via the landed `PickerField` (default `AppConfig.input_dir`), listing the resolved GDE files as FileChips + an `extract_required_files` missing-file warning. The Streamlit upload model was a browser constraint; the Flet app runs on the district server with real filesystem access (like the CLI + the nightly run), so a folder pick is the natural, parity-consistent model and reuses the landed picker.
6. **The diff (`_compute_diff`) + quality report.** *Decided: keep the quality report* (one existing `DataQualityReport().analyze(...).to_text()` call, a collapsible — cheap, high-value); *defer the diff* (a collapsible nice-to-have, not core to the verdict; a ROADMAP note if field-wanted). Avoids gold-plating.
7. **SFTP: re-transform vs upload-only leg.** *Decided at Spec (leaning upload-only):* to avoid re-running the pandas transform for the delivery leg, `convert_job` (or a sibling `upload_job`) writes-then-uploads from the frames retained in the view after a successful build; the Spec pins whether the delivery reuses retained `outputs`/`field_orders` (cheaper, no re-transform) or re-runs the job with `sftp_requested=True` (simpler, one code path). Both are correct; the Spec picks the simplest that doesn't re-transform needlessly.
