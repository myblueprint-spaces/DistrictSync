## Problem
Home is the flagship UX surface of the whole rebuild. DistrictSync's real engine is an invisible nightly **scheduled CLI** sync; the Flet app is its cockpit for a non-technical admin who opens it 2–3×/year, whose deep job is **trust — "I want to stop worrying that the roster synced."** Home must answer *"is my sync OK?"* in one plain-language **verdict** (green/amber/red) **before** any metrics, and it must **never crash** — a cockpit that throws a stack trace destroys trust.

Today the Flet `home` slot is either the IA-2 onboarding hero (unconfigured only) or the branded placeholder (configured). There is **no health dashboard**: a configured admin who opens Home learns nothing about whether last night's sync ran, delivered, or broke. The Streamlit surface being ported (`Home.py`) shows only a static "Configured / No active schedule" banner and never reads the run log for a verdict; worse, its status read is wrapped in a bare `try/except Exception` (line 72) that **silently hides the broken state** — precisely the failure `0013:43` calls out ("the state Streamlit's bare try/except hid"). The IA model (`0013:43`) specifies **Home branches THREE ways**: (a) unconfigured → onboarding hero + Start Setup; (b) configured + healthy → green verdict + metric tiles; (c) configured-but-broken → explicit amber/red "needs attention" naming the fault + a fix path — with status reads **degrading gracefully to a calm "status unavailable"**, never a crash.

The trust-critical decision — *given the run history + config + schedule state, what is the verdict and what does it say* — has **no home** and is untested. And the `__DISTRICTSYNC_RUN__` parser lives inline in the Streamlit page (`03_Run_History.py`), coupled to Streamlit and unreusable; IA-6 (Run History) will need the same parser. IA-3 builds both as **pure, tested, neutral modules** (the trust heart) plus the three-way view that renders them.

## Goals / Non-goals
- **Goal — a PURE, tested status-derivation module (COUNTED — the trust heart):** given parsed run records + `AppConfig` state + schedule state, derive a `Verdict` (HEALTHY/WARNING/FAILED) + a plain-language headline + supporting detail + a **fix path** for the attention/failed cases. Rules are explicit + defensible (see *Derivation rules* below). **Graceful degradation is a first-class OUTPUT:** unreadable/unparseable inputs → a calm "status unavailable" verdict, never a raise. The pure derivation is SEPARATE from I/O (reading/parsing the log) so it is unit-testable on synthetic records with zero filesystem.
- **Goal — port the run-log reader/parser to a pure, tested, neutral module:** reads the run-log file(s) and yields structured records (newest-first). This is IA-3's first consumer; **IA-6 reuses it** — design for that reuse (not coupled to Home). Trust-critical → COUNTED + tested (valid tags parsed, malformed lines skipped, missing file → empty, newest-first ordering).
- **Goal — the three-way Home view (coverage-omitted):** a dispatcher rendering (a) `build_onboarding(...)` when unconfigured; (b) a **verdict-first healthy** dashboard (green `HealthVerdictBanner` + humanized last-run summary + light metric tiles); (c) a **broken** dashboard (amber/red `HealthVerdictBanner` naming the fault + a concrete fix-path CTA). Verdict-first, humanized (friendly district name; **plain** timestamps — never a raw ISO/filename/path/stack trace).
- **Goal — build DS-1's deferred empty/loading state patterns here (first consumer):** the **empty** state (configured but no runs yet → a calm "no runs yet — your first nightly sync will appear here", NOT a scary red) and a **loading** pattern *iff* the status read is async (it is not — see *Sync/async decision* — so loading is a documented not-built pattern, not gold-plated).
- **Goal — wire into the shell:** replace IA-2's unconfigured-only `home` swap with a **three-way dispatcher**; branch (a) reuses `build_onboarding` (no throwaway — IA-2's plan promised this); (b)/(c) are the new dashboard.
- **Non-goal — the full Run History table (IA-6) or Convert (IA-5):** Home only *summarizes* the latest run + *links* to Run History. The parser is built reusable so IA-6 doesn't rebuild it; the staleness helper IA-6 also wants (`0013:64`) is scoped minimally here (Home needs staleness for verdict (c)) and IA-6 reuses it.
- **Non-goal — changing how run records are produced.** ZERO change to `src/etl|config|sftp|scheduler|main.py`. We READ the log, config, and (optionally) task state; we do not touch the emitter. Asserted as a gate.
- **Non-goal — a charting library / rich metrics.** Light metric tiles (entity counts, plain last-run time, SFTP delivered ✓), not visualizations.

## Derivation rules (the explicit, testable decision list)
The pure `derive_home_status(records, app_config, *, now=None) -> HomeStatus` returns a small frozen result: `verdict: Verdict`, `headline: str`, `detail: str`, `fix: FixAction | None` (a plain-language CTA label + a target destination id, e.g. `("Open Setup", "setup")` / `("Check Run History", "run_history")`), and `metrics: HomeMetrics | None` (entity counts + plain last-run time + sftp-delivered flag, populated only when a latest run exists). `records` is the newest-first list from the parser (may be empty). Evaluated **top-down, first match wins** (order encodes precedence — attention/failure never masked by a later "healthy"):

1. **Unconfigured** → the dispatcher renders branch (a) onboarding *instead of* calling derivation (derivation is not asked for a verdict when `not app_config.is_complete()`). Documented here so the boundary is explicit: derivation assumes a configured install.
2. **Status unavailable (graceful degradation, calm not alarm)** — if `records is None` (the reader signalled an unreadable/parse-catastrophe sentinel — see parser contract) → `WARNING` tone, headline *"Sync status unavailable"*, detail *"We couldn't read the run log right now — your nightly sync may still be running normally."*, fix → *"Check Run History"* (`run_history`). **This is the never-crash floor.** (An empty-but-readable list is case 3, not this.)
3. **No runs yet (empty, configured)** — `records == []` and configured → the **empty state** (DS-1 pattern built here): a calm `WARNING`-toned-but-reassuring banner, headline *"No sync has run yet"*, detail *"Your first nightly sync will appear here after it runs"* (+ if `schedule_registered`: *"— scheduled for {plain schedule_time} each night."* else a nudge to finish scheduling). Empty ≠ error (never red). Fix → *"Finish setup"* (`setup`) when `not schedule_registered`, else no fix (just wait).
4. **Last run failed** — latest record `status != "success"` → `FAILED` (red). Headline *"Last sync failed"*, detail names the fault in plain words from the record's `error` (humanized — never a raw traceback; truncate/sanitize), fix → *"Check Run History"* (`run_history`). Precedence over SFTP/data-errors (a failed ETL is the dominant fault).
5. **SFTP delivery failed** — latest `status == "success"` but `sftp_attempted and not sftp_ok` → `FAILED` (red — the roster did NOT reach SpacesEDU; mirrors CLI exit 3 + `_status_cell`). Headline *"Your roster didn't reach SpacesEDU"*, detail *"The data was built but the upload failed."*, fix → *"Check Run History"* (`run_history`).
6. **Anomaly / >20% drop** — latest success + delivered, but `anomalies` non-empty → `WARNING` (amber). Headline *"Something looked off in the last sync"*, detail summarizes the anomaly count in plain words (e.g. *"One roster file was smaller than usual."* — never the raw `ANOMALY:`-prefixed string), fix → *"Check Run History"* (`run_history`). Below hard-failures (5), above data-errors (7) — a delivered-but-suspicious run is attention, not failure.
7. **Data errors present** — latest success + delivered, no anomaly, but `data_errors.total > 0` → `WARNING` (amber). Headline *"Completed with {n} data warnings"*, detail *"A few records had field problems and were skipped — the sync still delivered."*, fix → *"Check Run History"* (`run_history`).
8. **Stale (last good run older than expected)** — latest record is a clean delivered success but its `timestamp` is older than a **staleness threshold** derived from `schedule_time` (a nightly job → expect a run within ~the last ~26–48h; pin the exact window in the Spec, generous to timezone/skew) → `WARNING` (amber). Headline *"No recent sync"*, detail *"The last successful sync was {plain relative time} ago — a nightly run may have been missed."*, fix → *"Check Run History"* (`run_history`). Staleness is the ONE rule needing `now` + `schedule_time`; the helper (`is_stale(last_ts, schedule_time, now)`) is pure + tested + **reused by IA-6** (`0013:64`).
9. **Healthy** — latest run is a recent, clean, delivered success with no anomalies/data-errors → `HEALTHY` (green). Headline *"Your roster is syncing"*, detail *"Last sync delivered cleanly {plain relative time}."*, no fix, metrics populated (entity counts + plain last-run time + delivered ✓). The reassurance the whole surface exists to give.

**Totality / degradation invariants (TESTED):** every branch returns a valid `HomeStatus` — no path raises. A record missing an expected key (partial/old schema) is read defensively (`.get(...)` with safe defaults), never `KeyError`. A record with an unparseable `timestamp` degrades the staleness check to "cannot determine staleness" (skip rule 8, do not crash) — the run still classifies on its other fields. If derivation itself hits an unexpected error, the **view** wraps the render in the `ErrorCard` fallback (see view section) so even a derivation bug shows a calm surface, not a stack trace.

## Parser contract (reusable by IA-6)
New pure module `src/ui_flet/run_log.py` (neutral name — NOT `home_*`; IA-6 imports it). No `flet` import. Contract:

- `read_run_records(log_path: Path | None = None) -> list[dict] | None`
  - Resolves the log path from `user_log_file()` when `log_path is None` (the `log_path` arg is the **test seam** — point it at a fixture file). Consults the same canonical path the Streamlit page + CLI use (single source of truth); the legacy `etl_tool.log` relative fallback is NOT reproduced (dead for the Flet era — note in the Spec).
  - **Missing file → `[]`** (empty, readable — *not* an error; drives derivation rule 3 "no runs yet").
  - **Present file → parse every line containing `__DISTRICTSYNC_RUN__`**, taking the JSON after the tag (`line.split(TAG, 1)[1].strip()` → `json.loads`); **malformed / non-JSON lines are SKIPPED** (mirrors `03_Run_History.py:60`), never raising.
  - **Returns records NEWEST-FIRST** (reverse file order — the file appends chronologically; Home wants latest at `[0]`). This is the reuse-shaping decision: IA-6 renders the list; Home reads `records[0]`.
  - **Unreadable file (permission/OS error on open/read) → `None`** — the graceful-degradation sentinel that drives derivation rule 2 ("status unavailable"). Distinguished from `[]` (missing = no runs yet vs. present-but-unreadable = can't tell). This split is the load-bearing contract; both are TESTED.
- Keep it minimal (no pandas, no DataFrame — a `list[dict]`): Home reads `records[0]`; IA-6 renders the whole list. A `TAG = "__DISTRICTSYNC_RUN__"` module constant documents the coupling to the (unchanged) emitter.

## Sync vs. async status-read decision (RECOMMENDED: synchronous, on mount)
**Read the log synchronously when the Home view is built.** Rationale:
- The FLET conventions' worker-thread rule (`docs/FLET_1.0_CONVENTIONS.md` §"Worker-thread → UI marshalling") exists for **`run_pipeline`** — a blocking pandas ETL that must never freeze the window. Reading `~/.districtsync/etl_tool.log` is a **small local text file** (kilobytes; a nightly-appended run-log), parsed line-by-line into a `list[dict]` — microseconds-to-low-milliseconds, not a freeze risk. Marshalling it onto a worker thread + `page.run_task` would add real concurrency surface (the #1 correctness trap per the conventions doc) for **no user-perceptible gain** — a clear YAGNI on the async path.
- Therefore the dispatcher calls `read_run_records()` + `derive_home_status(...)` **inline in the factory** and renders the result. No thread, no loading spinner needed on first paint.
- **Consequence for the "loading state" goal:** because the read is synchronous, there is **no async gap to cover** — the loading-state pattern is documented as *deliberately not built* (there is nothing to load asynchronously), NOT gold-plated with a speculative skeleton. If a future surface (IA-5 Convert's live run) needs a loading state, it builds it at that first real consumer (build-at-first-consumer, matching DS-1's own deferral rationale). The **empty** state IS built here (rule 3) — it is a real, reachable state.
- The only failure a sync read can hit (an unreadable log) is already a **modeled OUTPUT** (parser → `None` → derivation rule 2 "status unavailable"), so "graceful degradation" is delivered by the pure pipeline, not by async error handling.

## The three-way Home view
New `src/ui_flet/screens/home.py` (view, coverage-omitted): `build_home(page, *, app_config, on_navigate) -> ft.Control` — a **callback-driven factory** mirroring `onboarding`/`nav_rail` discipline (owns no lifecycle; `on_navigate(dest_id)` is injected by the shell = `select_by_id`). Dispatch:

- **Branch (a) unconfigured** — `not app_config.is_complete()` (or not `schedule_registered`, matching the IA-2 predicate) → return `build_onboarding(page, sis_type=app_config.sis_type, on_start_setup=lambda: on_navigate("setup"))`. **Reused verbatim** — IA-2's factory drops in, honouring its "IA-3 reuses onboarding as branch (a)" promise. No throwaway.
- **Branches (b)/(c) configured** — read `records = read_run_records()`, compute `status = derive_home_status(records, app_config)`, and render **verdict-first**:
  - The `HealthVerdictBanner(status.verdict, headline=status.headline, detail=status.detail)` FIRST (green healthy / amber attention / red failed — the answer to "is my sync OK?" before any metric).
  - When `status.metrics` is present (healthy / delivered runs): a light **metric tiles** row (built from `components.card` — NOT a charting lib): entity counts (Students/Staff/Family/Classes/Enrollments + the two myBlueprint+ counts when non-zero), **last run time in PLAIN language** (via a humanize timestamp helper — "last night at 3:00 AM" / "2 days ago", never the raw ISO), and **SFTP delivered ✓** when `sftp_ok`. Humanized district name in the header (`friendly_district_name`).
  - When there's a **fix path** (`status.fix`): a prominent `components.primary_button(status.fix.label, lambda _e: on_navigate(status.fix.dest_id), ...)` under the verdict — the concrete "Open Setup" / "Check Run History" CTA branch (c) requires.
  - The empty state (rule 3) renders the calm banner + a "Finish setup" or "waiting for first run" message — never red.
- **Never-crash wrapper (the reliability floor):** the whole configured-branch body is wrapped so that any unexpected exception during read/derive/render returns `components.ErrorCard("We couldn't show your sync status", "Your nightly sync keeps running in the background.", action=<Check Run History>)` instead of propagating — the cockpit shows a calm error surface, never a stack trace. (Derivation + parser are already total; this wrapper is defense-in-depth for the view layer, matching the `ErrorCard` intent DS-1 shipped for exactly this.)

## Risks & mitigations
- **The status read crashes Home (the #1 trust risk)** → the failure is a **modeled output**, not an exception path: parser returns `None` on an unreadable log → derivation rule 2 returns a calm "status unavailable" `WARNING`; the view additionally wraps read/derive/render in an `ErrorCard` fallback. All degradation axes are TESTED. The cockpit never shows a stack trace.
- **Derivation rules mis-rank a fault (e.g. show green when SFTP failed)** → the rules are a **first-match-wins ordered list** with failure/attention above healthy; each rule + its precedence is a COUNTED test on a synthetic record. Ported faithfully from `03_Run_History._status_cell` (the proven axes: status → sftp → data_errors) + extended with anomaly + staleness + empty.
- **Staleness false-positives on timezone/skew** → the threshold is generous (a nightly job → ~26–48h window, pinned in Spec) and staleness is `WARNING` (amber "no recent sync"), never `FAILED`; an unparseable timestamp **skips** the staleness rule (classifies on other fields) rather than crashing.
- **Parser coupling to the emitter drifts** → `run_log.py` reads the record defensively (`.get` with defaults), so an added/removed emitter field doesn't break parsing; the `TAG` constant + a Spec note document the (unchanged) contract. Zero emitter change (gate).
- **Sync read blocks the UI** → rejected as a real risk: a kilobyte local log parsed to a `list[dict]` is not a freeze; the async path is YAGNI (adds the conventions doc's #1 concurrency trap for no gain). Documented in *Sync/async decision*.
- **Onboarding branch regresses IA-2** → branch (a) calls `build_onboarding` UNCHANGED with the same `sis_type` + an `on_start_setup` that navigates to `setup`; the IA-2 manual smokes (hero renders, Start-setup navigates, district greeted) are re-run.
- **PII / raw-string leak on the broken card** → the `error`/`anomaly` strings are humanized/sanitized before display (plain summary, never a raw traceback/path); privacy is an explicit view constraint + a manual-smoke check.
- **Coverage** → `run_log.py` + `home_status.py` + the `humanize` additions COUNTED (carry the 80% gate); only `screens/home.py` added to the omit list (matches the established `shell/nav_rail/components/picker_field/setup/onboarding` view-omit pattern).
- **Zero core touch** → no `src/etl|config|sftp|scheduler|main.py` diff; `run_log` only *reads* the log path (`user_log_file`) + `home_status` only *reads* `AppConfig` — architect core-untouched gate holds.

## Test strategy
- **Unit (COUNTED) — `tests/test_ui_flet_run_log.py`** (`read_run_records`, via the `log_path` fixture seam over a temp file):
  - **valid tags parsed:** a fixture log with N `__DISTRICTSYNC_RUN__` lines → N records, each a dict with the expected keys.
  - **newest-first:** records appended chronologically → `read_run_records()[0]` is the LAST-written run (assert on `timestamp`/a marker field).
  - **malformed lines skipped:** a file mixing valid tag lines, a tag line with non-JSON, and untagged noise → only the valid records returned, no raise.
  - **missing file → `[]`** (a path that doesn't exist → empty list, the "no runs yet" signal).
  - **unreadable file → `None`** (simulate an open/read `OSError` — e.g. a directory path or a monkeypatched `open` raising → the "status unavailable" sentinel). The `[]`-vs-`None` split is asserted explicitly.
- **Unit (COUNTED) — `tests/test_ui_flet_home_status.py`** (`derive_home_status` on SYNTHETIC records — no filesystem — + `is_stale`):
  - **each rule, first-match precedence:** synthetic latest-record fixtures for — failed ETL (rule 4 → `FAILED`); success+SFTP-failed (5 → `FAILED`, precedence over data-errors); success+delivered+anomaly (6 → `WARNING`); success+delivered+data_errors (7 → `WARNING`); success+delivered+stale (8 → `WARNING`); clean recent delivered success (9 → `HEALTHY`, metrics populated). Assert verdict + a headline/fix invariant (e.g. failed → fix targets `run_history`).
  - **empty (rule 3):** `records == []` + configured → the calm empty `WARNING`-not-red state; `schedule_registered` toggles the "scheduled for {time}" vs "finish setup" fix.
  - **status unavailable (rule 2):** `records is None` → "Sync status unavailable" `WARNING`, never a raise.
