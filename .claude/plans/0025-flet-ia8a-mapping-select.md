## Problem
Mapping is the **"which roster mapping is DistrictSync using, and how do I see or switch it?"** surface. A district's field mapping (which SIS columns map to which SpacesEDU CSVs) is authored as a YAML config today; the admin picks one at first-time Setup. But there is currently **no dedicated surface to REVIEW that choice** — to answer "what config am I on, what does it actually produce, and how do I switch to a different pre-built one without re-running the whole Setup wizard." The Streamlit Mapping Editor's "Start" step offers exactly this list-and-pick affordance (buried as step 1 of a 7-step wizard whose remaining 6 steps are the deferred IA-8b editor).

Today the Flet `mapping` destination is a **branded placeholder** (`build_placeholder`, `shell.py:118`); the real config-editing surface lives only in Streamlit `04_Mapping_Editor.py` (deleted at CUT-1). Per the scope-lock, the **full editor is DEFERRED to ROADMAP** (the config-capability epic) — the admin authors district configs for now. So IA-8a is **NOT** a port of the editor; it is the **select-a-pre-built-config sliver**: a surface to see the active config, understand what it produces, and switch to another bundled one.

The design tension this plan must resolve: **Setup's folders step already has a district dropdown** (`_district_options` → `AppConfig.sis_type`). So what does a separate Mapping surface add that isn't a redundant second dropdown? This plan resolves it (see below): the Mapping surface is the **review-and-change** home for the district config — an admin who wants to *see* their current mapping and *what it emits*, or switch it, goes to **Advanced → Mapping**, not back through first-time Setup. It reuses — never duplicates — Setup's selection logic.

## The Mapping-vs-Setup reconciliation (the CENTRAL call)
**RECOMMENDATION: shape (a) — Mapping is a dedicated REVIEW + CHANGE surface for the district config, NOT a duplicate dropdown.** It adds an **informational, read-only summary of what the selected config produces** (its friendly district name, the human-readable list of output CSVs it emits from `enabled_entities`, and the count of source GDE files it reads) that Setup's bare dropdown does NOT surface, plus a select/apply control. It reuses Setup's `available_configs`/`friendly_district_name` option logic (DRY), never copies it. Justification:

- **Setup and Mapping answer different questions.** Setup is the **first-run "get me configured"** flow (folders + district + schedule + SFTP, all on one scroll) — the district dropdown there is one field among many, chosen once, with no explanation of what each config *does*. Mapping is the **ongoing "review and switch my mapping"** surface (Advanced group) — its whole job is to make the district-config choice *legible*: which config is active, **what CSVs it produces**, and a calm switch. An admin returning months later to check or change their mapping should not have to re-open the first-run Setup wizard (with its schedule/SFTP sections) to do it. This is the standard IA split: Setup = onboarding; Advanced/Mapping = the settings home for one specific concern.
- **The added value is the SUMMARY, not the dropdown.** A bare second dropdown *would* be redundant. The Mapping surface earns its place by answering **"what does this config actually produce?"** — the plain-language output-CSV list (`Students, Staff, Family, Classes, Enrollments` for a SpacesEDU district; `+ Courses, Student courses` for a myBlueprint+ tier), derived from `enabled_entities`. Setup never shows this; an admin picking `sd48myedbc` vs `mbp_core` from Setup's bare list has no idea the latter drops the 5 rostering CSVs. Mapping makes that consequence visible *before* applying. That is a genuine product-ux gain, not a duplicate.
- **The selection logic is REUSED, not duplicated.** Both surfaces need "list the configs, label each by its friendly district name." That logic is already single-sourced: `available_configs()` (the enumerator) + `friendly_district_name()` (the id→name map, TOTAL). Setup's `_district_options()` composes them into `ft.dropdown.Option`s. This slice does NOT re-implement enumeration or naming — it calls the same `available_configs()` + `friendly_district_name()`. **[design-Q: extract Setup's `_district_options` into a shared helper now that Mapping is a 2nd consumer, or call the two primitives directly? Recommendation: call `available_configs` + `friendly_district_name` directly in the new catalog module (they ARE the shared single source); do NOT extract the tiny `ft.dropdown.Option`-building `_district_options` into `components` — the two surfaces render the choice differently (Setup: a plain dropdown field; Mapping: a summary-carrying selector), so the shared seam is the two pure primitives, not the flet-coupled option builder. Promote `_district_options` to shared only if a 3rd consumer needs the identical `ft.dropdown.Option` list — the harness "promote on Nth use" rule. See design-Q 1.]**
- **Apply writes the SAME `AppConfig.sis_type` Setup writes.** Applying a Mapping selection persists `cfg.sis_type = chosen; cfg.save()` — byte-identical to what Setup's folders Save does for that field (`setup.py:141-142`). It does NOT re-write `input_dir`/`output_dir` (those stay as Setup left them) and does NOT touch schedule/SFTP. `sis_type` is a UI-owned runtime setting (not core), already written by Setup — so this is not a new write surface, just a second, focused writer of the same field. A validity note: switching `sis_type` on an install whose folders aren't set yet leaves `is_complete()` still false (harmless — the admin finishes folders in Setup); switching it on a configured install just re-points the mapping the next run uses. The apply surfaces a verdict-first confirmation naming the newly-active district.

**The gate rules.** If the gate prefers shape (b) — e.g. "fold this into Setup and drop the Mapping destination" — the fallback is noted in [Design questions](#design-questions-i-had-to-decide-surfaced-for-the-gate), but the recommendation, grounded in the onboarding-vs-settings IA split + the summary's genuine added value + the scope-lock naming Mapping an unconditional surface, is **shape (a): a dedicated review-and-change surface**.

## The config-catalog derivation (the COUNTED pure module)
**There IS real, useful, testable logic here** — the "what does this config produce" summary — so a pure module is warranted (not manufactured). New module `src/ui_flet/mapping_catalog.py` (PURE, no `flet` import, COUNTED):

```python
@dataclass(frozen=True)
class ConfigSummary:
    sis_type: str            # the raw id (secondary technical hint only, never the primary label)
    district_name: str       # friendly_district_name(sis_type) — the primary label
    output_labels: tuple[str, ...]   # plain-language CSV names, in a stable order (what it emits)
    source_file_count: int   # how many distinct GDE source files it reads (0 if underivable)
    loaded_ok: bool          # False when load_config raised → this is a safe degraded summary

def list_configs(config_dir: Path | None = None) -> list[ConfigSummary]: ...
def summarize_config(sis_type: str, config_dir: Path | None = None) -> ConfigSummary: ...
```

- **`list_configs`** = `available_configs(config_dir)` → `summarize_config(id, config_dir)` for each → a `list[ConfigSummary]` (already sorted by `available_configs`). The `config_dir` seam is passed straight through (test seam, mirrors `friendly_district_name`), so it's unit-testable against a fixture mappings dir with no home dependency.
- **`summarize_config`** = `load_config(sis_type, config_dir)` then derive:
  - `district_name` = `friendly_district_name(sis_type, config_dir=config_dir)` (TOTAL — the friendly label, or the raw id if the config has no `district_name`).
  - `output_labels` = the enabled entities mapped to plain CSV names. **Resolution (single-sourced with the pipeline):** `enabled = list(gc.enabled_entities) if gc.enabled_entities else list(cfg.mappings.keys())` (the SAME empty-means-all rule as `models.py:310-314`), then each entity key → its friendly label via the entity-label map (design-Q 2), emitted in a **stable canonical order** (rostering entities first, then myBlueprint+ — reusing `home_status._ROSTERING_ENTITIES`/`_MYBLUEPRINT_ENTITIES` as the ordering spine so the label order matches Home/Run History). An unknown/non-standard entity key falls back to the key itself (total).
  - `source_file_count` = the count of **distinct** source filenames across the enabled entities' `EntityConfig.source_files.values()` (a de-duped set — the same file often feeds several entities). A read-only "this config reads N extract files" fact; `0` if none derivable.
- **Total / defensive (reliability-resilience):** `summarize_config` wraps `load_config` in `try/except`. A config that **fails to load** (raises `FileNotFoundError`/`ValueError` — e.g. a partner-authored YAML in `~/.districtsync/mappings/` with a validation error) returns a **safe degraded `ConfigSummary`** (`district_name = friendly_district_name(sis_type)` which itself falls back to the raw id, `output_labels = ()`, `source_file_count = 0`, `loaded_ok = False`) — **never a crash**. `list_configs` therefore always returns one summary per enumerated id, some possibly degraded. The view renders a degraded summary calmly ("we couldn't read this config's details" — see the view section), and the apply control for a `loaded_ok = False` config is **disabled** (you cannot switch TO a broken config — `run_pipeline` would `sys.exit(1)` on it; gate the apply structurally, mirroring Setup's Save-gate discipline).
- **Privacy note (LIVE/top):** a `ConfigSummary` carries only config **structure** — the district name, output-CSV labels, a file count. It carries **no student PII** (the config is a column-name mapping, not data) and does **not** interpolate a raw exception string into any admin-facing field (a load failure is named by category, not by echoing the Pydantic error). This is asserted in the test strategy (a planted-broken-config case checks the degraded summary carries no raw error text).

**Why this is a real module, not a manufactured seam (honesty clause):** unlike IA-7 (which honestly had no pure logic — two constants + a launch), IA-8a has genuine derivation: the empty-`enabled_entities`-means-all resolution, the entity→CSV-label mapping in a canonical order, the de-duped source-file count, and the total-over-a-failing-config degradation. That is trust-surface logic an admin's switch decision depends on, testable without flet — it belongs COUNTED, exactly like `run_history.to_run_rows`/`home_status.derive_home_status`.

## The select + apply flow (the view)
`src/ui_flet/screens/mapping.py` (VIEW, coverage-omitted) — `build_mapping(page, *, app_config) -> ft.Control`, a factory mirroring `run_history.py`/`home.py`:

3. **A "Switch mapping" control** — a selector of the available pre-built configs (`list_configs()`), each labelled by friendly district name + its output-CSV summary line, so the admin sees **what each produces before choosing** (the product-ux gain). 4. **An "Apply" action** — `components.primary_button("Use this mapping", …)`, **structurally disabled** until a config *different from the current* AND `loaded_ok = True` is selected (mirroring Setup's Save-gate: you can never apply a broken config or a no-op). Owns no lifecycle; **sync build on mount** — `list_configs()` reads a handful of small local YAMLs in microseconds (the same justification as Run History's sync read; the worker-thread convention is scoped to `run_pipeline`), so no async.

## Design questions I had to decide (surfaced for the gate)
1. **Extract Setup's `_district_options` into a shared helper (now a 2nd consumer), or reuse the two underlying primitives?** → **Reuse the primitives (`available_configs` + `friendly_district_name`) directly in `mapping_catalog`; do NOT extract the flet-coupled `_district_options`.** *Decided: DRY at the right seam + the harness "promote on Nth use" rule.* The genuinely shared logic is enumeration + naming, and that is ALREADY single-sourced (`available_configs`/`friendly_district_name`). `_district_options` is a thin `ft.dropdown.Option`-builder coupled to Setup's plain-dropdown render; Mapping renders the choice differently (a summary-carrying selector), so copying or extracting the option builder would share the wrong thing. Both surfaces converge on the two pure primitives — the correct shared seam. Promote `_district_options` to `components` only if a 3rd surface needs the identical `ft.dropdown.Option` list.
2. **Where does the entity-key → friendly-CSV-label map live?** → **A small module-level map in `mapping_catalog.py`, keyed on the same entity tuples `home_status` already defines.** *Decided: single source of truth + KISS.* The entity KEYS (`Students`/`Staff`/…/`CourseInfo`/`StudentCourses`) are already canonical in `home_status._ROSTERING_ENTITIES`/`_MYBLUEPRINT_ENTITIES` (reuse those for ORDER). The human CSV LABELS (e.g. `CourseInfo` → "Courses", `StudentCourses` → "Student courses", matching `components.run_table`'s column headers `components.py:235-238`) are a presentation fact the catalog owns; define them once in `mapping_catalog` (not scattered). If a 2nd consumer ever needs the same label map, promote it to a shared pure module (e.g. into `humanize` or a new `entities.py`) — not this slice.
3. **Switch control: a dropdown + live summary, or a list of selectable summary cards?** → **A `ft.Dropdown` (keyed by `sis_type`, labelled by `friendly_district_name`) mirroring Setup's district field, with the chosen option's full `ConfigSummary` rendered in a panel beneath it.** *Decided: product-ux consistency + KISS.* It reuses the familiar Setup idiom, keeps the surface calm (no long scroll of cards), and still shows the "what it produces" summary for the *pending* choice before Apply — the key legibility gain. A list-of-cards is a heavier layout with no added value for a handful of configs. Pin the exact control at Spec (the recommendation is the dropdown).
4. **Shape (a) dedicated review-and-change surface vs shape (b) fold into Setup / drop the Mapping destination?** → **Shape (a).** *Decided: the onboarding-vs-settings IA split + the summary's genuine added value + the scope-lock naming Mapping an unconditional Advanced surface.* Setup is the first-run flow; Mapping is the ongoing settings home for the district-config concern, made legible by the output-CSV summary Setup never shows. Folding into Setup would force an admin back through the whole first-run wizard to review/switch one setting, and would drop a destination the scope-lock keeps. *(Fallback if the gate prefers (b): drop `screens/mapping.py`, keep `mapping_catalog.py` as the summary source, and surface the summary + switch inside Setup's folders card — but this loses the settings-home separation and is not recommended.)*
5. **Thread `on_navigate` into `build_mapping`?** → **No (minimal signature `build_mapping(page, *, app_config)`).** *Decided: KISS + the IA-6 precedent.* Mapping is a self-contained settings surface with no in-app fix-path CTA to another screen (Apply stays on-surface with a verdict banner) — mirroring `build_run_history`'s minimal signature. Add `on_navigate` only at a future consumer's need (build-at-first-consumer).
6. **Show the raw `sis_type` at all?** → **Yes, but only as a small secondary technical hint, never the primary label.** *Decided: product-ux (plain-language first) + a pragmatic escape hatch.* The friendly district name is the primary label everywhere; the raw id (`sd48myedbc`) appears as a muted caption so a support interaction ("which config are you on?") has the exact id — without ever making the admin read a machine id as the main text. Matches the humanize-first rule while keeping the id recoverable.
