## Problem

Shan's 2026-07-08 field test of the released v3.4.0 Flet UI surfaced ~12 verified root-cause defects plus a product-level gap: the UI was ported for feature parity (Plan 0013) and repeatedly **asserts state it never verified**. DistrictSync is a trust instrument — a district admin must believe the nightly sync is working without watching it. All claims below carry file:line evidence and were independently adversarially verified:

1. **Stale AppConfig snapshot** — `shell.py:170` loads `AppConfig` once at startup and binds the instance into Home/Run History/Mapping/Help via `functools.partial`; screens rebuild per navigation but from the frozen snapshot. District switch never propagates until restart; Mapping's own "current mapping" card stays stale (`mapping.py:124` vs `_on_apply` at `mapping.py:165` which loads a *separate* fresh instance); the Apply gate (`mapping.py:145-147`) compares against the stale value so a switch cannot be reverted without restart; Home's onboarding branch (`home.py:143`) never leaves "not set up yet" after in-session setup. Setup/Convert already load fresh (`setup.py:94`, `convert.py:147`) — the pipeline always used the right config; display/trust defect, not data corruption.
2. **Run history store is not production-grade** — Run History parses `__DISTRICTSYNC_RUN__` lines from the rotating diagnostic log (`run_log.py:58` reads only the live `etl_tool.log`; 5MB×3 rotation silently erases history; RotatingFileHandler is not multi-process-safe against the scheduled CLI writer on Windows). **pytest pollutes the real user log**: `src/main.py:47` runs `get_logger` at import time and `tests/conftest.py` has no path isolation — 365 of the 370 records in Shan's live log are test fixtures. Free-text `error=str(e)` (paths/columns) is persisted (`pipeline.py:244`). Manual Convert runs never appear at all (`convert_job` bypasses `run_pipeline`/`_emit_run_log` by design — `convert.py:102`).
3. **Schedule state divergence** — `setup.py:303` persists `schedule_time` + `schedule_registered` **before** `register_task` runs and never reverts on failure; status everywhere (`home_status.py:257`, `nav.py:74`, Run History footer) trusts the config boolean and never reconciles with the real task (`query_task` is dead code that also omits `/V` — `windows.py:544-565`). Live proof: config says "scheduled 15:36" while the task was externally deleted (Event 141). "Save setup" (`setup.py:133`) silently ignores the schedule section. No unregister affordance (`delete_task` has zero callers). Task action path is pinned to wherever the exe ran from (e.g. `Downloads\`) with no warning — and when the pinned exe is gone, the task fires and fails **without ever writing a run record** (a blind spot only the task's own LastTaskResult can see).
4. **Elevation dead-end** — unattended registration (Password logon + `RunLevel Highest`, `windows.py:260-266`) genuinely requires an elevated caller; the only affordance is "quit and re-open as administrator" (`setup_errors.py:57-61`).
5. **SFTP test trust gaps** — the test is real (live paramiko auth verified) but: it **writes a typed password to the keyring before testing** (`setup.py:587-588`) so a typo'd password clobbers a working credential even when the test fails; it tests live unsaved form fields (`setup.py:471-477`) while the nightly uses saved config (`pipeline.py:447-451`); success copy (`setup.py:628-629`) never names host, user, or credential source.
6. **Nav model** — state-dependent destination reordering (`nav.py:100-114`) destroyed spatial memory and read as instability; programmatic navigation never moves the rail highlight (`shell.py:215-217`; `build_nav` discards the rail reference, `shell.py:14-15`), desyncing all three `on_navigate` call sites.
7. **Silent district defaults** — `AppConfig.sis_type` defaults to `"myedbc"` (`app_config.py:30`), pre-selected in Setup (`setup.py:170`); Convert falls back to `configs[0]` alphabetically (`convert.py:253`).
8. **Convert output dead-end** — success view (`convert.py:396-422`) never shows where files were written; the PII "never render a raw path" rule (`convert_result.py:15-20`) is over-applied to a local, app-owned config path. Empty `output_dir` silently falls back to the *input* dir (`convert.py:147`).
9. **Chrome/input craft** — Exit is a no-op: Flet 0.85.3 `Window.destroy()`/`close()` are `async def` (`flet/controls/core/window.py:355/367`); `shell.py:223`, `shell.py:243` (latent), `launcher.py:104` call them synchronously. No `page.window.icon`, no brand `.ico`, no `--icon` on either `flet pack` (Makefile:37-76, `.github/workflows/flet-pack.yml`). No `TextField` wires `on_submit`.
10. **Onboarding triple-door** — while unconfigured: Setup-led nav order + a Setup tab + a Home onboarding hero all point at one job.

## Goals / Non-goals

- **Goal (product bar):** the UI avoids asserting states it can't confirm; **every success names WHAT it checked and WHEN** (e.g. "connection to <host> tested just now", "task is registered — next run tonight at HH:MM") rather than promising future outcomes; "where am I" and "where are my files" always answerable; setup ends in an honest, checked finish line; run history is a clean, district-scoped, durable ledger; professional craft baseline (brand icon, Enter works, Exit works, stable nav).
- **Goal:** first-run Setup is a stepped wizard (Folders → District → Schedule → Delivery → checked summary) that graduates into a flat Settings surface (user-approved direction). Schedule and Delivery steps are **skippable** ("set up later") so the aha moment isn't gated on an admin password + live SFTP credential being at hand.
- **Goal:** schedule registration self-elevates per-operation via a normal UAC prompt (user-approved); the app stays non-admin.
- **Goal:** dedicated run store (SQLite, per-user app-data dir) with hard test isolation, replacing log-parsing; ScheduledTasks read-back as the primary schedule signal **when readable**, with an honest "can't confirm right now" state when not.
- **Non-goal:** YAML mapping editor (ROADMAP); Linux/macOS elevation parity; any change to ETL transform semantics or output CSV contracts (SD74 snapshot byte-identical); multi-user/server features; auto-update; push alerting on failed/missed nightly runs (→ ROADMAP — the panel's premise-challenge: a pull cockpit can't alert an admin who doesn't open it; genuinely valuable, separate egress/PII surface); visual design-system redesign (tokens/theme/components stay).

## Approach

**One combined redesign program** (user-chosen). Trust-critical logic continues the established split: pure COUNTED modules (tested) + thin view glue (coverage-omitted). Design decisions (2c-revised after the 10-advisor panel):

- **D1 — Config freshness via load-per-build.** Screens rebuild per navigation; bind `AppConfig.load` (a supplier) instead of a loaded instance — the pattern Setup/Convert already use. Reactive store rejected (YAGNI). Mapping's Apply writes through + re-renders its own surface; its gate compares against the *persisted* current value. Freshness on navigation only; a manual **Refresh** affordance on Home/Run History covers the Watcher who leaves the app open overnight (cheap; re-runs the surface build).
- **D2 — Run store = SQLite (`history.db`) in `paths.user_data_dir()`.** Written via one shared `write_run_record()` seam called from **both** `run_pipeline` and `convert_job` (the manual path must produce rows too). Hybrid schema: promoted typed columns for filtering (`timestamp` TEXT naive-local-ISO (matches existing records), `sis_type`, `source` CHECK-constrained ∈ {manual, scheduled, cli, unknown}, `status` CHECK-constrained, `error_category` bounded enum, `schema_version` INTEGER) + one `record` JSON column holding the **exact flat record dict the derivation modules already consume** — so `home_status`/`run_history` need zero shape changes and exactly one reader exists. Composite index `(sis_type, timestamp)`. `PRAGMA user_version = 1`; **no migration framework now** (YAGNI) — additive-only schema rule recorded in DECISIONS, all statements name columns explicitly (no `SELECT *`/positional INSERT), and a writer seeing `user_version` higher than it knows **does not migrate/downgrade** (two exe versions share this DB: the pinned scheduled exe + an updated UI). WAL + `busy_timeout=5000` (WAL = readers concurrent with a writer, writers serialized — busy_timeout makes the rare UI-write-during-nightly-write wait, not error); DELETE-journal fallback if WAL unavailable. Open-use-close per operation (no shared connection; UI reads on UI thread, Convert writes on the worker thread). Reader contract mapped explicitly: missing DB → `[]` (read path never creates the DB), 0 rows → `[]`, any `sqlite3.Error`/`OSError` → `None`. Write path is the only creator/migrator; on "database disk image is malformed" it quarantines (`history.corrupt-<ts>.db`) and recreates, logging the event (a torn write must not brick the ledger forever). **No compaction** (≈1.8k rows/~400KB per year — YAGNI; retention → ROADMAP if ever real). **No backfill** (panel consensus: the only source is the already-polluted log — 365/370 test fixtures; importing it would seed the clean ledger with garbage). Instead, **honest fresh-start empty states**: the store records its own `created_at`; when the store is fresh but the install is not (config exists / schedule registered), Run History/Home say "Run history starts fresh with this update — earlier runs aren't shown", never "No sync has run yet".
- **D2a — Log-line parity (ops).** The `__DISTRICTSYNC_RUN__` line **stays** in the diagnostic log and gains the same enrichment (source, sis_type, error_category): the record dict is built once and written to both sinks. The log keeps the rich free-text error/anomaly detail (ops needs "KeyError: final mark" at 2am); the store carries only the bounded `error_category` + counts (privacy split). Store lifecycle events (WAL fallback taken, quarantine/recreate) are logged INFO/WARNING. `docs/partner/troubleshooting.md` documents `history.db`, the log's ops role, and the unchanged exit-code contract.
- **D2b — Store writes are best-effort and strictly non-fatal, positioned after output commit.** Wrapped try/except at both call sites; a store failure (locked, disk full, corrupt) logs WARNING/ERROR and never alters `PipelineResult`, the CSVs, or the exit-code contract (0/1/2/3); the failure-path write must never raise and mask the original ETL exception (`pipeline.py:425-426`). The enriched log line is the durable fallback when the store write fails.
- **D2c — `source` propagation:** the registered task's action injects `DSYNC_SOURCE=scheduled` (single-sourced with the action builder in `register_task`); CLI default `cli`; `convert_job` passes `manual`. Without the marker every nightly run would mislabel — the field's whole purpose is "did the NIGHTLY sync run".
- **D3 — Test isolation is a correctness fix.** Autouse conftest fixture patches the **load-bearing deep seam `paths.user_data_dir`** (patching `user_log_file` alone is a no-op for `logger.py`, which imported the name by reference at module top — `logger.py:4`), AND covers `AppConfig` — which today bypasses `paths.py` entirely via import-time `APP_CONFIG_DIR = Path.home()/".districtsync"` (`app_config.py:19-20`): refactor AppConfig to resolve through `paths.user_data_dir()` at call time (single source; also unblocks any future relocation). The store resolves its path at call time via `paths.user_history_db()` — never a module-level constant (or every pipeline/e2e/SD74 test would write the real `history.db`). Fixture also swaps the keyring backend (no test touches the real Credential Manager) and restores/close global logging handlers on teardown (a leaked FileHandler blocks tmp_path cleanup on Windows). A **canary test** deliberately calls `AppConfig.save()` + `get_logger()` + a store write under the fixture and asserts the real log AND config.json AND history.db are untouched. `get_logger` moves out of import time and is explicitly called in **every** entry path (CLI dispatch + Flet launcher), each covered by a test. Honest register: this is *redirect-the-seams + guard tests* — it holds while write paths route through the patched seams; the canary is the tripwire, not a mechanical impossibility.
- **D4 — Schedule truth via a ScheduledTasks-module reader, tri-state.** New reader combines `Get-ScheduledTask` (exists / state / action path) + `Get-ScheduledTaskInfo` (NextRunTime; LastTaskResult captured from the same query) — `Get-ScheduledTaskInfo` alone cannot report existence or action path. Reader reuses the register-script hardening verbatim: FIXED script text referencing only `$env:DSYNC_TASKNAME` (never interpolated), name through `validate_task_name` first, `-EncodedCommand`, datetimes emitted culture-invariantly (ISO round-trip), bounded `subprocess` timeout. Contract is **three-state** and owned by a new pure COUNTED module `src/ui_flet/schedule_status.py`: `LIVE` (task found; next run known) / `MISSING` (definitively queried and absent → re-register route) / `UNKNOWN` (PowerShell missing, timeout, access denied — e.g. an elevated-registered task unreadable by the filtered token). **Only MISSING may claim "not scheduled"; UNKNOWN renders "we couldn't confirm the schedule right now" and never falls back to asserting "scheduled" from the config boolean.** All consumers (Home verdict, Setup readout, Run History footer/empty-state copy, nav badge) consume this one typed derivation — the read-back result is fetched once per surface build **off-thread** (existing `page.run_thread` pattern, "checking the schedule…" placeholder), injected into the pure derivation; `nav.py` stays subprocess-free. `schedule_status` also detects the "task fired but no run record exists" contradiction via LastTaskResult (the Downloads-pinned-exe blind spot) and logs a WARNING when read-back contradicts the config flag. Config writes move **after** registration success; Unregister affordance added (idempotent: deleting an absent task presents as "already not scheduled"; may itself require the D5 elevation for an elevated-registered task — same path). Registering from a transient location (Downloads/Temp) warns before pinning.
- **D4a — Two distinct facts, never conflated:** `setup_completed` (durable config fact: the wizard finish line was reached at least once — folders + district committed; schedule/delivery possibly deferred) vs `schedule live now` (D4 read-back). **Wizard entry gates on `not setup_completed` only.** A previously-configured install whose task broke lands on **Home with a WARNING + a fix path routed to Setup's schedule section** — never back into first-run onboarding (the Event-141 Firefighter must not be greeted as a newcomer). `nav.needs_setup` is redefined accordingly and all four current consumers of `schedule_registered` move to the appropriate fact.
- **D5 — Per-operation elevation, isolated in `src/scheduler/elevation.py`** (SRP: a generic privilege-elevation IPC primitive, not task-scheduler logic; `windows.py` consumes it). `ShellExecuteExW(lpVerb="runas")` on the **absolute System32 `powershell.exe` path** (elevation raises PATH-hijack stakes) with `SEE_MASK_NOCLOSEPROCESS` → `WaitForSingleObject` with a **bounded timeout** (never INFINITE; on timeout: terminate child, calm degradation) + `GetExitCodeProcess`. Inbound (password-bearing) channel: DPAPI **CurrentUser**-scope encrypted JSON blob (structured JSON, not KEY=VALUE — paths can contain `=`) with app-scoped optional entropy, owner-only ACL set explicitly, in `paths.user_data_dir()` (one app-data location); **fail closed on SID mismatch** (over-the-shoulder UAC under a different admin account cannot decrypt CurrentUser DPAPI — that is correct; never widen to LocalMachine scope, which would downgrade a domain credential's confidentiality; surface "elevation ran as a different account — log in as an administrator or use the no-password schedule"). Outbound result: **plaintext** JSON (it carries no secret — a sanitized ok/message) written **atomically** (temp + rename); parent handles child-exited-but-result-missing/partial by degrading to "couldn't confirm registration" and re-deriving via D4 read-back. The elevated child's message passes the same `_clean_ps_stderr` + `DSYNC_`-strip sanitization before surfacing or storing; the script keeps the never-echo-`$env:DSYNC_TASK_PW` property. Best-effort startup sweep deletes orphaned handshake files. Whole flow runs off the UI thread with visible states: in-flight ("waiting for the Windows permission prompt…"), declined (`ERROR_CANCELLED` 1223 → calm copy), accepted-but-failed (child crashed / result missing / registration error → through `classify_schedule_error`), timeout. UAC-success is verified by the D4 read-back, not assumed from exit code alone.
- **D6 — SFTP test side-effect-free and provenance-honest:** `test_connection(password_override=...)` threads a typed password transiently to `client.connect()` only — never keyring, never logs, never the returned message; a test asserts `store_password` is not called on the test path. Keyring written only on explicit Save. Success copy names what was tested: "Connected to `<host>` as `<user>` using the password saved in this computer's credential manager" / "…using the password you just entered — click Save to keep it". Unsaved host/user edits soften the claim: "these settings work — Save to use them for the nightly sync" (present-tense, test-scoped; never "the nightly sync can deliver" for unsaved values).
- **D7 — Nav: fixed order always** (Home, Convert, Run History, Setup, Mapping, Help); newcomer guidance via initial selection (launch into the wizard while `not setup_completed`) + a "needs attention" badge on Setup driven by `schedule_status` (lands with the schedule-truth slice, where its signal exists). `build_nav` exposes the rail; `select_by_id` syncs the highlight — one change fixes every programmatic hop. Rail label stays "Setup" (spatial memory); the graduated surface titles itself **"Settings"** with a one-time transition cue after wizard completion.
- **D8 — Wizard→Settings hybrid**, pure state machine in `src/ui_flet/setup_flow.py` (five concrete named steps — **no data-driven step engine**, YAGNI). `setup_flow` is genuinely pure: it consumes **injected verification results** (`folders_valid`, `schedule_state: LIVE/MISSING/UNKNOWN`, `delivery_tested: ok/failed/skipped`) — the view/boundary layer performs all I/O and feeds results in. Steps persist to their existing sinks as they commit (folders/district → AppConfig, schedule → task then config, SFTP → keyring+config); **resume derives the first unsatisfied step from real state — no stored cursor** — and **reconciles against side effects already performed** (read-back for a task the Schedule step already registered; keyring for a stored credential) showing "already scheduled/stored" instead of double-registering. No single step may flip `setup_completed`; the finish line is the only completion signal, so mid-wizard abandonment never reads as "set up". Back navigation + "Step 2 of 5" indicator; Enter advances when the step gate is satisfied; focus moves to the new step's first field. Per-step async states enumerated (schedule: UAC in-flight/declined/failed/timeout; finish: read-back tri-state; delivery: testing/ok/failed/skipped). **Finish-line copy is honest and adaptive** — it names what was checked and when, never guarantees the future: SFTP configured → "You're set up. Tonight at HH:MM DistrictSync will build <district> and try to deliver to SpacesEDU — we tested the connection to <host> as <user> just now and it worked."; SFTP skipped → "…will build <district> into your output folder. Set up delivery whenever you're ready."; schedule skipped → "…Run conversions from the Convert tab; add a nightly schedule whenever you're ready." Settings mode: **one** Save that reconciles — re-registering the task when **any field baked into the task action changed** (input_dir, output_dir, sis_type, sftp flag, run_time — not time only), with the live "Currently registered: daily at HH:MM — next run <t>" readout beside it.
- **D9 — Explicit district everywhere:** `sis_type` default `""`; "Choose your district" placeholder; auto-select only when exactly one config exists; Convert refuses to run without an explicit district (no `configs[0]` fallback). Existing installs unaffected.
- **D10 — Convert output visibility within the PII rule:** `ConvertResult` stays path-free; the *view* renders the resolved output folder + "Open folder" (`os.startfile`); resolved output dir shown on the form before running; empty `output_dir` blocks with a routed "set your output folder in Setup" instead of silently writing into the input folder.

Alternatives rejected: whole-app `--uac-admin` (PII posture, mapped drives, breaks scheduled launch); runs.jsonl (no concurrent-writer story); reactive config store (YAGNI); log filter-at-read (rotation loss + pollution remain); "restart as admin" relaunch (loses typed state; session stays elevated); backfill (source data is polluted); compaction + migration framework + data-driven wizard steps (YAGNI, panel-confirmed); DPAPI-encrypting the outbound result (no secret in it).

## Architecture & holistic fit

- **Codebase fit:** preserves the three-layer split and COUNTED-pure vs view-glue convention. New modules: `src/history/store.py` (+ `__init__.py`) — data layer, UI-neutral and platform-neutral (pipeline writes, UI reads; no `ui_flet` import); `src/scheduler/elevation.py` — Windows elevation IPC primitive consumed by `windows.py`; `src/ui_flet/schedule_status.py` — the single pure owner of schedule-status derivation + precedence (mirrors `home_status.LatestReason`; `query_task`'s replacement returns a typed frozen dataclass, not `dict[str,str]`); `_friendly_schedule_time` relocates here from `home_status` (cohesion). `src/ui_flet/setup_flow.py` — pure wizard state machine; the Setup **view** decomposes into per-step render helpers + a settings-scroll renderer (no god-view). Exactly **one** run-record reader post-migration: the store reader reconstructs the existing flat-dict record shape; `run_log.py` retires (its `[]/None` contract and tests migrate to the store reader). No new dependencies (stdlib `sqlite3`, `ctypes`).
- **Product fit:** direct implementation of the Discover deliverable (trust instrument; Installer/Watcher/Firefighter personas — D4a exists precisely so the Firefighter is never treated as an Installer). Personas + journey states persist to `docs/claugentic-PRODUCT.md` at program start (not the last slice) so per-slice review uses the persona lens.
- **Quality dimensions to uphold:** `data-and-persistence` (store contract, additive-only schema, quarantine recovery, non-fatal writes — including its privacy/PII facet: bounded error_category in the store, rich detail only in the local log, per CLAUDE.md's top-priority student-PII scope), `reliability-resilience` (tri-state read-back, elevation timeout/orphan cleanup, store degradation), `security` (DPAPI scope fail-closed, System32 pin, sanitization across the elevation boundary, keyring-on-Save-only, validators on all new subprocess surfaces — including the secret/PII-leak facet of the elevation handshake), `observability-ops` (log-line parity, store lifecycle logging, exit-code contract provably unchanged, troubleshooting docs), `product-ux` (per-step states, honest empty states, fix routing, keyboard flow), `testing` (isolation canary, failure-path fixtures, honesty about manual-Verify surfaces), `maintainability-structure` (single-source schedule status, one reader, SRP module placement).
- **Future-proofing:** `user_version` + additive-only rule keeps a v2 schema open without building migration machinery; `source` tag enables later filtering UI; `schedule_status` gives Linux read-back a slot; elevation primitive is reusable for a future elevated Unregister. Not building now: history export, retention, push alerting (ROADMAP), pluggable wizard steps.
