    • 2026-06-25 — Plan 0011: graceful UI shutdown — idle watchdog + Exit controls + single-instance guard, gated by a write_guard over the atomic loader (INVARIANTS candidate). The frozen UI exe ran sys.exit(stcli.main()) → the Streamlit server + console leaked forever after the browser tab closed (no built-in "exit on last disconnect"), and the leaked port-8501 occupant blocked relaunch. New src/ui/lifecycle.py concentrates the fix: a daemon idle watchdog (started once per process via @st.cache_resource in Home.py) reads the live browser-session count and os._exit(0)s after a 90s disconnect grace (keys off websocket-gone, not inactivity — an open-idle tab stays alive); explicit Exit DistrictSync (sidebar, shared brand.sidebar_exit_control()) + wizard Finish & Close buttons call request_exit(); and launcher.already_running(8501) opens a browser to an existing instance and exits instead of starting a second server. Three load-bearing dependencies recorded (re-verify on any streamlit bump): (1) the watchdog reaches the PRIVATE Runtime._session_mgr.num_active_sessions() — verified on streamlit 1.54.0, isolated to ONE helper (lifecycle._active_session_count) so a rename fails loudly in one place, wrapped in try/except that degrades to a logged no-op (never crashes the UI), and pinned streamlit>=1.54,<1.55 in requirements.txt + pyproject.toml [ui]/[dev] (requirements-dev.txt inherits via -r); (2) the single-instance guard keys off the /_stcore/health body == ok (not bare connectivity/200 — a non-Streamlit 8501 occupant won't match); (3) write_guard()/safe_to_exit() is the write-integrity gate — 02_Convert.py wraps its SFTP upload path (DataLoader.save_all into a TemporaryDirectory + upload_csvs) in write_guard(), and BOTH the watchdog (defers to next tick) AND request_exit() (bounded ~30s wait) refuse to exit while a write is in flight, so an exit can't truncate the in-flight SFTP upload into a partial remote delivery. Corrected at architect-review: the UI's save_all stages into a TEMP dir, not the real output dir, so it cannot tear data/output — the guard's real value is the SFTP egress, plus defense-in-depth for any future direct-to-output UI write (whose finally-based atomicity a hard os._exit would skip). should_exit (pure), write_guard/safe_to_exit, and already_running are unit-tested (tests/test_ui_lifecycle.py) despite the src/ui coverage omit; a ui-marked read-only smoke (no os._exit) asserts the private reach stays live. webbrowser.open opens a NEW tab to the existing instance (can't focus the existing tab) — accepted. launcher.already_running relies on streamlit's transitive requests dependency (bundled via --collect-all streamlit) — re-verify on a streamlit bump. CLI/scheduled-task path untouched; SD74 snapshot byte-identical (UI is outside the ETL path).

    • 2026-06-25 — Plan 0009 Slice 2: scheduler errors de-CLIXML'd + the wizard classifies registration failures by elevation. A real failed Register-ScheduledTask surfaced PowerShell 5.1's CLIXML error serialization — a #< CLIXML … <Objs>…<S S="Error"> : Access is denied.</S></Objs> blob that echoes the whole failing script (incl. the $env:DSYNC_TASK_PW literal) and buries the human message — because Write-Error to a redirected stderr is CLIXML-wrapped. Fix at the source: the script's catch now emits the bare message via [Console]::Error.WriteLine($_.Exception.Message); exit 1 (plain text, no CLIXML; success path Write-Output 'DSYNC_OK' unchanged). Defensive Python fallback: _clean_ps_stderr(text) runs on captured stderr BEFORE the canonical-marker match — if #< CLIXML is present it extracts ONLY the first <S S="Error"> node (the message line; later nodes echo the script body, never returned), decodes _x000D_/_x000A_, strips the leading "<cmd> : " positional prefix; otherwise returns the text stripped. Bounded regex over <S S="Error">(.*?)</S> (no XML parser on untrusted text); a leak test asserts the cleaned message contains "Access is denied" but neither CLIXML/<Objs/the script body nor DSYNC_TASK_PW/the secret. So the canonical markers ("PowerShell not found" / "ScheduledTasks module not available" / Windows' "Access is denied") still match the cleaned text and register_task returns a clean one-liner. Wizard: new is_elevated() in windows.py (win32 ctypes.windll.shell32.IsUserAnAdmin() wrapped try/except → False; non-win32 → False; module-level + mypy-clean since UI is mypy-excluded). The wizard's blanket "Run as administrator" failure branch is replaced by a pure, unit-testable _classify_schedule_error(msg, elevated): not-elevated + access-denied → run-as-admin; elevated + access-denied → batch-logon-right / wrong-password (account password not Windows Hello PIN / MSA microsoft.com password) hint; "PowerShell not found" / "ScheduledTasks module not available" → their own actionable lines; else → the clean msg. Kills the misleading run-as-admin loop the already-elevated SD54 admin hit. Slice-1's leak tests still pass against the new catch. UX-only — no SD54 dependency; SD74 snapshot byte-identical (scheduler outside the ETL path).
    • 2026-06-25 — Plan 0009 Slice 1: Windows scheduled-task registration moved from schtasks /Create /XML to PowerShell Register-ScheduledTask. The 2026-06-15 long-path fix paired a /XML-declared <LogonType>Password</LogonType> with schtasks /RU /RP credentials; that handoff fails ERROR: Access is denied even when elevated (the SD54 rollout blocker). Fix: register_task now runs a FIXED PowerShell script (_build_register_script(has_password), references only $env:DSYNC_*) by base64-encoding it as UTF-16LE and passing it via powershell.exe -EncodedCommand (no script file on disk) and passes all dynamic values — incl. the password — via a fresh-copy child env (_build_env(), {**os.environ, …}; os.environ never mutated). -EncodedCommand, not stdin (-Command -): piping a multi-line try/catch to -Command - over stdin silently no-ops on PowerShell 5.1 (read line-by-line, the block returns exit 0 with no output and registers nothing → reports success while doing nothing); -EncodedCommand parses the script as one complete unit so the fail-loud try/catch actually runs. Do not revert to stdin. The unattended path forces the stored-credential logon with an explicit New-ScheduledTaskPrincipal -LogonType Password -RunLevel Highest (not the -User/-Password/-RunLevel parameter-set inference, which can silently degrade to S4U/Interactive on some PS 5.1 builds) → New-ScheduledTask -Principal → Register-ScheduledTask -InputObject $task -User -Password -Force. Password-via-env, not argv is a hardening win over the legacy /RP <pw> (which was visible to all users in the process list); the script never echoes $env:DSYNC_TASK_PW, and a leak-closure test asserts the value + the DSYNC_TASK_PW literal appear in neither the returned message nor caplog (success and failure). Settings parity with the old XML preserved (-StartWhenAvailable:$false no-catch-up, battery flags, PT2H, IgnoreNew); run time parsed with InvariantCulture (locale-safe). No-password path → -LogonType Interactive -RunLevel Limited (matches the prior InteractiveToken, never S4U; run_highest stays ignored). Fail-loud canonical messages — "PowerShell not found" (FileNotFoundError) / "ScheduledTasks module not available" (cmdlet-not-found) / else raw PS exception text verbatim — are the contract the Slice-2 wizard classifier keys off. delete_task/query_task deliberately stay on schtasks.exe (read-only/name-only, no length/credential surface, fully tested, and query_task's LIST-parse output is consumed by Home.py) — consistency migration is a tracked ROADMAP follow-up, not this slice. No new PyInstaller hidden imports (plain subprocess to a system binary). Implementer verified the no-password path live (registers Interactive, queried, deleted); the credentialed/unattended path is gated on the user live-verification checklist (LogonType=Password/RunLevel=Highest + logged-off SFTP-reachable run + Last Run Result=0x0).     • 2026-06-24 — Plan 0008 Item 3 (revised, post-approval): stale entity CSVs are ARCHIVED after each run, not deleted. Added DataLoader.archive_stale_outputs(emitted) -> sorted[str]: detects stale entity CSVs via the pure detect_stale_outputs(emitted) (lists <EntityName>.csv whose stem is a registry-known entity and not emitted this run), then moves each into a lazily-created output_dir/archive_<ts>/ via os.replace (atomic same-fs); run_pipeline calls it after a successful save_all and logs an INFO with the archived names (no exit/run-log change). Non-destructive os.replace move sidesteps the cross-config data-loss that blocked an auto-delete: under _base inheritance no delete allow-list is cleanly safe — a registry/mappings.keys() set would erase a DIFFERENT config's inherited-but-disabled entity CSV sharing the dir (e.g. an mbp_core-written CourseInfo.csv), and an enabled_entities set would BOTH miss a config-dropped entity AND wipe an enabled entity's last-good CSV on an empty-input run — so moving aside (recoverable bytes) is the safe primitive. Enabler: the SFTP uploader globs output_dir.glob("*.csv") — top-level only, non-recursive (uploader.py:182) — so a file inside the archive_<ts>/ SUBfolder is auto-excluded from delivery (a stale CSV can no longer ship). Best-effort: a per-file move failure is logged at ERROR and skipped (the roster already committed + delivered — an archive hiccup must not fail the run). Supersedes the earlier warn-only landing. Foot-gun (Stage-9): any future output-dir delete/prune must still derive from the resolved enabled_entities, never mappings.keys() — _base inheritance defines more entities than a config enables, so mappings.keys() includes inherited-but-disabled entities and would cause cross-config data loss.
    • 2026-06-23 — Plan 0007: save_all commit is now true all-or-none (os.replace + per-file .bak_<ts>/ backup/restore). Replaced the per-file shutil.move commit loop — non-atomic across files (a move failing on file K left output a torn mix of new 1..K-1 + stale K..N, then the staging dir was deleted so the tear was permanent) and (Windows overwrite) within a file (shutil.move's os.rename raises FileExistsError on an existing dest → copy2+unlink fallback can leave a half-written dest). New _commit_staged(sorted(staged_files), backup_dir): per file, move any existing target aside into .bak_<ts>/ then promote the staged file with os.replace (atomic same-fs overwrite); on any failure roll back in reverse applied order (remove new dest / restore backup) and re-raise the original — rollback completes before save_all's finally rmtrees .bak_<ts>/ (restore-before-cleanup invariant). New-entity path takes no backup and rolls back via unlink(missing_ok=True), never os.replace(None, …). Reproduce-first: 4 role-based os.replace-injection tests captured the torn output as a failure first. SD74 byte-identical. (Chose Option A over keep-shutil.move+backup, which leaves the within-file tear unfixed.)
    • 2026-06-23 — Plan 0006: the "fail loud, don't fake success" trio (one slice). Three "false green" defects in an unattended PII tool fixed together. (1) apply_field_map is now row-resilient + loud. Previously series.apply(func) aborted the WHOLE column if any one row raised, and the outer except Exception silently blanked it to pd.NA while the run reported success — and (corrected fact, verified at runtime) the unknown-transform raise ValueError was ALSO swallowed by that same handler (returned [<NA>, <NA>], only logged a traceback), so it never actually failed loud. New: transforms apply per-row — a row whose func raises gets pd.NA in that cell only while valid rows keep their correct value (so the accurate transform path is untouched → SD74 byte-identical); the unknown-transform / any column-level error blanks that column and continues (do NOT raise — config fail-fast validation of ALLOWED_TRANSFORMS is the cleaner fix, backlog T2.4). Every failure is recorded to the new TransformContext.data_errors ledger, logged at ERROR, and surfaced in the run-log as a data_errors summary ({total, by_field}) + Run History "Completed with N data errors". Data errors are a SEPARATE axis — ETL status stays success and the run still delivers (never silently swallowed, never fails the run for one bad row). TransformOutputs gains a third field data_errors; threaded out of run_transform via DataTransformer.data_errors. (2) No-usable-input guard in run_pipeline right after extractor.load_data: not raw_data or all(df.empty …) → ERROR naming the missing/empty files + raise → existing outer handler writes _emit_run_log("failed") + re-raises → main exit 1. Keys off INPUT presence (raw_data), not outputs=={} — chosen so a supported period-only sd51attendance run (period file present, daily absent) does NOT false-fail: outputs=={} would have, because run_transform's positional-primary skip drops StudentAttendance when the daily file (its listed-first source) is empty (a separate latent bug this guard neither masks nor fixes). A partial run with some empty sources stays exit 0 by design. (3) Run History honest Status (display-only, additive): the at-a-glance Status cell shows amber "ETL OK · SFTP FAILED" / "Completed with N data errors" derived from the same booleans already in each record (no exit-code or run-log status change; the dedicated SFTP ❌ column is unchanged). SD74 snapshot byte-identical (a clean run fires none of the new branches).

    • 2026-06-22 — Attendance Absence Date now emitted as ISO yyyy-MM-dd (was DD-MMM-YYYY). Reverses only the OUTPUT shape; rostering dates are untouched (only the standalone attendance feed was failing). Removed the now-dead format_dd_mmm_yyyy helper — a near-clone of normalize_iso_date differing only in its strftime (DRY / no dead code) — and swapped it for normalize_iso_date in ALLOWED_TRANSFORMS and the base StudentAttendance.field_map transform placeholder (that field_map's values are decorative: the transformer builds rows directly and never calls apply_field_map). ISO is a fixed SpacesEDU contract (not per-district), so it's hardcoded, not configurable (KISS/YAGNI). SD74 snapshot byte-identical (attendance not enabled there).
    • 2026-06-19 — Plan 0004: StudentAttendance.csv written WITHOUT the UTF-8 BOM. Root cause: DataLoader._write_csv wrote every CSV with utf-8-sig (BOM), and SpacesEDU's BOM-strict, case-sensitive header check read the first header as ﻿School Number ≠ School Number, so the file wasn't recognized and column mapping broke (mis-validating the date column). Fix: a DataLoader._NO_BOM_ENTITIES = {"StudentAttendance"} allowlist — those entities are written plain UTF-8 (no BOM); all other CSVs keep the utf-8-sig BOM so districts can still open them in Excel without mojibake. The rostering CSVs were unaffected because they reach SpacesEDU via the zip/OneRoster path, which tolerates the BOM; only the standalone attendance feed is strict. Regression test asserts StudentAttendance has no BOM and a rostering entity still does. (CRLF line endings kept — RFC-4180 standard; revisit only if SpacesEDU rejects them next.)

    • 2026-06-19 — Plan 0004 Slice 3: StudentAttendance 8-12 Student Period Absences band (per-period pass-through).     2026-06-15 — Windows scheduled-task registration via Task Scheduler XML¶

    SUPERSEDED-BY-0009 (2026-06-25). The schtasks /Create /XML + <LogonType>Password</LogonType> + /RU /RP credential handoff below regressed: it fails ERROR: Access is denied on the unattended (password) path even when elevated — the very "Verified LIVE … no iteration needed" claim that broke. Registration is now PowerShell Register-ScheduledTask (see the 2026-06-25 entry at the top). The /TR 261-char insight and the no-catch-up StartWhenAvailable=false rationale below still hold and are preserved in the new mechanism; the schtasks /XML + /RP mechanism is dead. delete_task/query_task remain on schtasks.exe. Retained for history.

    • Register via schtasks /Create /XML, not inline /TR. schtasks caps the /TR value at 261 chars; the source-mode command (cmd /c "cd /d "<repo>" && "<python>" -m src.main --sis … --input "…" --output "…" --sftp") hit 458 chars on this dev box and failed. XML has no length cap, sets <WorkingDirectory> natively (kills the brittle nested-quote cmd /c "cd /d …" wrapper), and carries the schedule/principal/run-level. Public API of register_task/query_task/delete_task/current_run_as_user unchanged; validation-before-OS-call and /RP password redaction preserved.
    • Password stays out of the XML; run-level moves into it. The run-as password is still passed via schtasks /RP on the command line (never written to the temp XML); /RU /RP /XML is a valid documented combination. The old /RL HIGHEST flag is replaced by <RunLevel>HighestAvailable</RunLevel> (password mode) / LeastPrivilege (no-password → <LogonType>InteractiveToken</LogonType>, matching the prior logged-on-only default).
    • XML written as UTF-16-with-BOM (encoding="utf-16" + <?xml … encoding="UTF-16"?>) — schtasks /Create /XML accepted it first try on Windows 11 (no iteration needed); UTF-16 chosen for cross-version compatibility. Temp file is always removed in a finally.
    • Hand-built XML string with xml.sax.saxutils.escape on every interpolated value (paths/user/args) rather than ElementTree — deterministic, testable output with explicit escaping at the boundary (validate-at-boundary). Daily trigger uses a fixed past StartBoundary date (2024-01-01) so the emitted XML is deterministic and never triggers a catch-up run (<StartWhenAvailable>false</StartWhenAvailable>).
    • Verified LIVE on this Windows 11 box end-to-end (register→query→delete) with a deliberately long input/output path: old /TR would have been 458 chars (>261) and failed; XML registered SUCCESS, queried Ready with the full command line intact, deleted cleanly.

    2026-06-05 — Zero-orphan enrollments (plan 0003, Slice 2)¶

    • Zero-orphan invariant via a published active roster. StudentTransformer publishes its filtered output's User ID set to context.active_student_ids (so active_student_ids == set(Students.User ID) by construction); classes.py (homeroom) and enrollments.py (homeroom + subject) then filter their student rows to that set. Result: no enrollment or homeroom class references a student absent from Students.csv. Verified on an SD48-shaped fixture: 3 orphans (2 homeroom + 1 subject) → 0. The single filter helper is BaseTransformer.filter_to_active(df, student_col, context, caller) — both consumers call it (DRY); chosen over re-deriving "is active" at each read site (would drift from the roster). Demographic Student Number and schedule Student ID share the pupil-number value space; both sides normalized with astype(str).str.strip().
    • Students-only filter; teacher/co-teacher rows untouched. Only the student-row derivations are filtered; teacher rows in _homeroom_enrollments/_subject_enrollments derive from the unfiltered frames, and _blended_teacher_enrollments/_classinfo_coteacher_enrollments are not touched — so teacher rows are byte-identical to before (regression-tested by holding the homeroom class set constant and comparing a mixed roster vs an all-active control). The one intended teacher-row change: a homeroom with only inactive students is dropped (no empty homeroom class) along with its teacher enrollment — the no-empty-homeroom goal, not a leak.
    • Never filter-to-empty (guard). When active_student_ids is empty (Students disabled / ran later, e.g. mbp_core) or the student column is absent, filter_to_active logs a WARNING and returns the frame unchanged rather than dropping every row. filter_to_active returns a .copy() so callers can mutate columns without a SettingWithCopyWarning (matches the other filter_* helpers).
    • SD74 snapshot byte-identical. SD74 has no inactive students, so its active set = all students → the filter is a no-op; the frozen snapshot stays green (the regression guard).
