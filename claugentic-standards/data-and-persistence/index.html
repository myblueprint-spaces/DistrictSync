    Data & Persistence — the stored state stays correct, consistent, and recoverable¶

    Loads when: a change touches the database — schema/models, migrations, queries, ORM mappings, repositories, transactions, indexes, or replication. Anything that reads or writes durable state. ISO/IEC 25010: Reliability (with Maintainability spillover on schema evolution) · Status: draft · v0.1.0

    Each entry below is one auditable dimension. Per change, the reviewer applies the relevant ones fully (select-don't-skip), right-sized to the change — never gold-plating an irrelevant one, never skipping a relevant one. The unifying invariant: the stored truth never silently corrupts, never silently loses a write, and can always be rebuilt.

    Schema design & normalization¶

    • Good looks like — The schema is normalized to ~3NF by default (no repeating groups, no partial/transitive dependencies); every column has a meaningful type and the tightest constraint that holds (NOT NULL, CHECK, UNIQUE, ENUM/lookup over free-text status). Denormalization exists only where deliberately chosen for a measured read path, is documented, and ships with a defined mechanism to keep the duplicated data consistent (trigger, transactional write, or materialized view with a refresh contract).
    • Auditor checks — [J] Is the model normalized, or is duplication accidental (same fact stored in two tables with no sync path)? [D] Do nullable columns that should be mandatory carry NOT NULL? [J] Are status/type columns constrained (enum/FK/check) rather than free strings? [J] For each denormalization: is there a stated read-path justification and a consistency mechanism, recorded in claugentic-DECISIONS.md?
    • Confidence — mixed (normalization and denormalization-justification are design calls [J]; column-constraint presence is gate-checkable via schema lint/diff [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Normalizing keeps each fact in exactly one place, so updates can't leave contradictory copies; the cost is more joins, which can be slower to read. Denormalizing copies data to make reads fast, but now you own keeping the copies in sync — skip that and the copies drift apart and lie.
    • Sources — Kleppmann, Designing Data-Intensive Applications, Ch. 2–3 (data models, normalization vs. denormalization); C. J. Date, An Introduction to Database Systems (normal forms).

    Indexing strategy¶

    • Good looks like — Indexes exist for the columns the workload actually filters, joins, and orders by — driven by real query predicates, not guessed. Composite indexes order columns by selectivity/usage (leftmost-prefix usable); covering indexes (INCLUDE/index-only scan) serve hot read-only queries without a heap fetch. Every foreign key has a supporting index on the referencing side. Redundant/unused indexes are removed (each one taxes every write).
    • Auditor checks — [J] Does each index map to a known predicate/join/sort in the codebase, or is it cargo-culted? [D] Does every FK column have a covering index (catch-able by schema introspection)? [J] For composite indexes, is column order justified by query shape (leftmost prefix)? [J] For a flagged hot read query, would a covering index avoid the table lookup? [D] Any duplicate indexes (same leading columns)?
    • Confidence — mixed (matching indexes to real query patterns and column ordering is analysis [J]; FK-index presence and duplicate-index detection are deterministic via schema introspection [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — An index is a pre-sorted lookup that turns a full-table scan into a direct hit, hugely speeding reads. But every index must be updated on every insert/update/delete and consumes storage, so unused indexes silently slow all writes for no benefit. Missing FK indexes make deletes and joins crawl as data grows.
    • Sources — Markus Winand, Use The Index, Luke! (use-the-index-luke.com — composite-index column order, covering indexes/index-only scans, leftmost-prefix rule); PostgreSQL docs §11 "Indexes"; Kleppmann Ch. 3.

    Migrations — versioned, reversible, zero-downtime¶

    • Good looks like — Every schema change is a versioned, ordered migration file in source control (not a manual prod edit), applied by a migration tool (Alembic/Flyway/Liquibase/Rails/Prisma/Django). Each migration is reversible (a real down, or an explicit documented note when irreversible). Schema changes that run against a live system follow expand-then-contract (a.k.a. parallel change): expand adds the new shape backward-compatibly → migrate backfills and dual-writes while old and new coexist → contract removes the old shape only once nothing reads it. No single deploy both adds a non-nullable column with no default and reads it. Long backfills are batched, not one giant locking statement.
    • Auditor checks — [D] Is the change a checked-in migration with a version/sequence (not ad-hoc SQL)? [D] Is there a down-migration, or an explicit irreversibility note? [J] For a destructive/blocking change on a live table (drop/rename column, add NOT NULL, type change), is it split into expand→migrate→contract across deploys rather than one breaking step? [J] Are backfills batched to avoid long table locks? [D] Does CI run migrations forward and back on a scratch DB?
    • Confidence — mixed (the zero-downtime sequencing is design [J]; migration-file presence, reversibility presence, and forward/back CI execution are deterministic gates [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Versioned reversible migrations mean every environment reaches the same schema the same way and a bad change can be rolled back. Expand-then-contract lets you reshape a database that's serving live traffic without a maintenance window — the cost is the change spans several deploys instead of one. Skip it and a "quick" rename takes the site down or strands old app versions mid-deploy.
    • Sources — Martin Fowler, "Parallel Change" (martinfowler.com/bliki/ParallelChange.html — the expand/migrate/contract framing); Sam Newman, Building Microservices, 2nd ed. (incremental, parallel database refactoring during decomposition); Alembic tutorial (alembic.sqlalchemy.org/en/latest/tutorial.html — versioned revisions, up/down); Flyway "Migrations" docs (documentation.red-gate.com/flyway/flyway-cli-and-api/concepts/migrations); Liquibase "Concepts" (docs.liquibase.com/concepts/introduction-to-liquibase.html); Rails "Active Record Migrations" guide (guides.rubyonrails.org/active_record_migrations.html).

    Transactions & isolation levels¶

    • Good looks like — A logical unit of work that must all-succeed-or-all-fail runs inside one transaction (the multi-step write, not a per-statement autocommit loop). The isolation level is chosen, not defaulted-by-accident: the team knows what the engine's default is (e.g. PostgreSQL/Oracle = Read Committed) and raises it (Repeatable Read / Serializable) for invariants that read-then-write across rows. Transactions are short (no network calls or user think-time inside an open transaction). Serialization failures under SERIALIZABLE are retried.
    • Auditor checks — [J] Are related writes wrapped in a single transaction so a mid-failure can't leave half-applied state? [J] Is the isolation level appropriate for the invariant — would the default's anomalies (non-repeatable read, phantom, write skew) break this logic? [J] Are transactions kept short (no I/O / external calls / waiting on a user while holding locks)? [J] Are SERIALIZABLE serialization-failure retries in place where used?
    • Confidence — judgment (transaction boundaries and isolation choice are design decisions tied to invariants; not mechanically gate-provable).
    • Tradeoff (plain English) — A transaction makes a group of changes atomic — all land or none do — so a crash never leaves the data half-updated. Higher isolation removes subtle concurrency bugs but reduces how many operations run at once and can force retries; the right level is the lowest one that still protects the specific rule you care about.
    • Sources — PostgreSQL docs §13.2 "Transaction Isolation" (Read Committed default; Repeatable Read prevents phantoms — stronger than the SQL standard; Serializable monitors for anomalies); Kleppmann Ch. 7 "Transactions" (weak isolation, write skew, phantoms); Berenson, Bernstein, Gray, Melton & O'Neil, "A Critique of ANSI SQL Isolation Levels" (SIGMOD 1995 / MSR-TR-95-51 — what the ANSI levels do and fail to specify).

    Concurrency control — optimistic vs. pessimistic locking (lost-update protection)¶

    • Good looks like — Any read-modify-write on shared rows is protected against the lost update (two transactions read the same row, both write, one silently overwrites the other). Either optimistic (a version/updated_at column checked on write; the UPDATE ... WHERE version = :v affecting 0 rows triggers a conflict the app handles/retries) or pessimistic (SELECT ... FOR UPDATE locks the row for the duration). The choice matches contention: optimistic for low-contention/read-heavy with long user think-time; pessimistic for hot, high-contention rows (inventory, balances). The naive "load object → mutate in memory → save whole row" path is never used on contended data without one of these.
    • Auditor checks — [J] For each read-modify-write on shared state, is lost-update prevented (version check or row lock), or can a concurrent writer be silently clobbered? [J] Does the strategy fit the contention profile (optimistic vs. pessimistic)? [D] If optimistic, is the conflict path handled (0-rows-affected detected and surfaced/retried, not ignored)? [J] Are pessimistic locks scoped tightly to avoid deadlocks (consistent lock ordering)?
    • Confidence — mixed (identifying read-modify-write hazards and picking a strategy is reasoning [J]; whether the optimistic conflict branch exists is gate-checkable in code [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Without this, two people editing the same record at once means the last save wins and the other's change vanishes with no error. Optimistic locking catches the clash at save time (cheap when clashes are rare, but you must retry); pessimistic locking blocks others up front (safe under heavy contention, but reduces concurrency and risks deadlocks).
    • Sources — Kleppmann Ch. 7 ("Preventing Lost Updates" — atomic ops, explicit locking, compare-and-set); PostgreSQL docs §13.3 "Explicit Locking" (SELECT FOR UPDATE); Martin Fowler, PoEAA (Optimistic/Pessimistic Offline Lock).

    N+1 queries & ORM pitfalls¶

    • Good looks like — Iterating a collection does not fire one query per element — related data is fetched in bulk via eager loading / joins / a batched IN (...) (e.g. selectinload/joinedload, prefetch_related/select_related, JOIN FETCH). The ORM's lazy-loading and identity-map behavior is understood; queries inside loops are flagged. Large result sets are streamed/paginated, not fully materialized into objects. ORM-generated SQL is inspected for hot paths (the abstraction is convenient, not magic).
    • Auditor checks — [J] Does any loop over rows access a lazy relation, triggering N+1? (Spot via query-count assertions in tests or echo/SQL logs.) [D] Can a regression test assert a bounded query count for the hot endpoint? [J] Are big reads streamed/chunked instead of loading every row into memory as objects? [J] Was the generated SQL for a critical path actually looked at?
    • Confidence — mixed (recognizing the N+1 pattern is reasoning [J]; a query-count assertion in a test is the deterministic backstop [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — ORMs make data access easy, which makes it easy to accidentally run one tiny query per item in a list — fine for 5 rows, catastrophic for 5,000. Loading relations in one batched query instead trades a little upfront code for an order-of-magnitude speedup; ignoring it produces endpoints that crawl as data grows.
    • Sources — Martin Fowler, PoEAA (Lazy Load, Identity Map); SQLAlchemy "Relationship Loading Techniques" (docs.sqlalchemy.org/en/20/orm/queryguide/relationships.html — selectinload, joinedload); Django "QuerySet API: select_related / prefetch_related" (docs.djangoproject.com/en/stable/ref/models/querysets/#select-related); Hibernate "Fetching strategies" (docs.jboss.org/hibernate/orm/current/userguide/html_single/Hibernate_User_Guide.html#fetching).

    Connection pooling & resource lifecycle¶

    • Good looks like — Connections come from a bounded pool sized to the DB's limits (pool size ≤ what max_connections and CPU can serve), never opened ad-hoc per request. Every connection/cursor/session is returned/closed deterministically (context manager / try-finally / framework-scoped), so failures don't leak handles. Pool timeouts, max-overflow, and recycle (to dodge stale/server-killed connections) are configured. Serverless/high-fanout deployments front the DB with a pooler (PgBouncer/RDS Proxy) instead of each instance holding its own pool.
    • Auditor checks — [D] Are connections acquired via a pool, not constructed inline per call? [D] Is every connection/session closed on all paths (context manager / finally)? [J] Is the pool sized against the DB's connection ceiling (won't exhaust it under peak concurrency)? [J] Is connection recycle/health-check set so stale connections don't surface as errors?
    • Confidence — mixed (pool sizing is a capacity call [J]; pooled-acquisition and deterministic-close are detectable in code/lint [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Opening a DB connection is expensive; a pool reuses a fixed set so requests are fast and the database isn't overwhelmed. Size it too small and requests queue; too big and you exhaust the database's connection limit and everything fails. Leaking connections (not closing them) slowly starves the pool until the app hangs.
    • Sources — PostgreSQL "Connection Settings" docs (max_connections — postgresql.org/docs/current/runtime-config-connection.html); HikariCP "About Pool Sizing" wiki (github.com/brettwooldridge/HikariCP/wiki/About-Pool-Sizing — the connections = ((core_count * 2) + effective_spindle_count) starting point); PgBouncer "Features" (pgbouncer.org/features.html — connection modes, pool sizing); AWS RDS Proxy docs (docs.aws.amazon.com/AmazonRDS/latest/UserGuide/rds-proxy.html — high-fanout pooling for serverless); SQLAlchemy QueuePool docs (docs.sqlalchemy.org/en/20/core/pooling.html#sqlalchemy.pool.QueuePool).

    Soft deletes & audit columns¶

    • Good looks like — Deletion strategy is a deliberate choice: hard delete where data should truly vanish (and for right-to-erasure/GDPR), soft delete (deleted_at/is_deleted) where history/recoverability/referential safety matter. Soft-delete is applied consistently — all default reads filter out soft-deleted rows (a base scope/queryset/view), uniqueness constraints account for it, and FKs don't dangle. Tables carry audit columns (created_at, updated_at, and created_by/updated_by where accountability matters), populated automatically (DB default/trigger or ORM hook), not by hand.
    • Auditor checks — [J] Is delete-vs-soft-delete chosen intentionally for this entity (and does soft-delete conflict with an erasure requirement)? [D] If soft-delete: do default queries exclude soft-deleted rows everywhere (no path leaks "deleted" data)? [J] Do unique constraints/business rules correctly treat soft-deleted rows? [D] Are created_at/updated_at present and auto-populated on the touched tables?
    • Confidence — mixed (the strategy and constraint interactions are design [J]; presence of audit columns and the default soft-delete filter are checkable [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Soft deletes keep a record around (marked deleted) so you can recover it and don't break things that referenced it — but every query must remember to hide them, or "deleted" data reappears, and it conflicts with legal "actually erase my data" requests. Audit timestamps cost nothing and answer "when/by whom did this change?" — omitting them makes incidents nearly impossible to reconstruct.
    • Sources — Martin Fowler, "Event Sourcing" (martinfowler.com/eaaDev/EventSourcing.html — capturing state changes as an append-only history rather than destructive overwrite); PostgreSQL docs on triggers/DEFAULT for auto-populated audit columns (postgresql.org/docs/current/sql-createtrigger.html); GDPR Art. 17 (right to erasure — tension with soft delete).

    Read replicas & replication lag¶

    • Good looks like — When reads are offloaded to replicas, the code accounts for replication lag (a replica can be milliseconds-to-seconds behind the primary). Operations that must see their own just-committed write — and any read-after-write the user expects to be consistent — go to the primary (or use read-your-writes routing), not a possibly-stale replica. Lag is monitored with an alert threshold; routing degrades sanely if a replica falls too far behind. Writes always go to the primary; the read/write split is explicit, not accidental.
    • Auditor checks — [J] Do read-after-write flows (user saves then immediately views) read from the primary or a lag-bounded path, avoiding stale data? [J] Is the write→primary / read→replica split explicit and correct (no writes sent to a read replica)? [D] Is replication lag monitored with an alert? [J] Does the system behave correctly if a replica is stale or unavailable (fallback to primary)?
    • Confidence — mixed (consistency-routing decisions are design [J]; lag monitoring/alerting is a deterministic ops check [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Read replicas multiply read capacity by serving copies of the data, but those copies lag slightly behind. If a user updates something and the next screen reads from a lagging replica, they see their old data and think the save failed. Routing must send "must-be-fresh" reads to the source of truth.
    • Sources — Kleppmann Ch. 5 (replication, replication lag, read-your-writes / monotonic-read consistency); PostgreSQL "Streaming Replication" (postgresql.org/docs/current/warm-standby.html#STREAMING-REPLICATION); PostgreSQL pg_stat_replication view for lag monitoring (postgresql.org/docs/current/monitoring-stats.html#MONITORING-PG-STAT-REPLICATION-VIEW).

    Query optimization (EXPLAIN, no SELECT *)¶

    • Good looks like — Hot/complex queries are validated with EXPLAIN/EXPLAIN ANALYZE — the plan uses indexes (index scan over seq scan on large tables), join strategies are sane, and there's no surprise full-table scan on a big table. Queries select only the columns needed (no SELECT *, which over-fetches, breaks covering indexes, and couples callers to column order). Pagination uses keyset/seek (WHERE id > :last) rather than large OFFSET on deep pages. Filtering/aggregation/limiting is pushed into the database, not done in app code after pulling everything back.
    • Auditor checks — [J] For a flagged hot query, does its EXPLAIN plan avoid a large-table seq scan and use the right index? [D] Is SELECT * absent from application queries (grep-able)? [J] Is deep pagination keyset-based rather than high-OFFSET? [J] Is work (filter/sort/aggregate) done in SQL rather than fetched-then-processed in memory?
    • Confidence — mixed (plan quality and pagination strategy are analysis [J]; SELECT * presence is a deterministic grep [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — EXPLAIN shows how the database will actually run a query, so you catch the one that scans a million rows before it hits production. SELECT * and big OFFSET paging feel convenient but pull back more than needed and get slower the deeper you page; naming columns and seeking by key keeps queries lean and stable.
    • Sources — Markus Winand, Use The Index, Luke! ("Use The Index, Luke" / "Slow Indexes", keyset pagination — "no offset"); PostgreSQL docs §14 "Using EXPLAIN"; Kleppmann Ch. 3.

    Referential integrity¶

    • Good looks like — Relationships are enforced by the database with FOREIGN KEY constraints (plus UNIQUE, CHECK, NOT NULL), not left to application code that a second writer or a script can bypass. ON DELETE/ON UPDATE behavior (CASCADE/RESTRICT/SET NULL) is chosen deliberately per relationship. Orphan rows are structurally impossible. Where a constraint is intentionally pushed to the app layer (e.g. cross-shard, or a DB that lacks FKs), that decision is explicit and the app enforces it on every write path, recorded in claugentic-DECISIONS.md.
    • Auditor checks — [D] Do relationship columns carry actual FK constraints in the schema (introspectable), or is integrity only hoped-for in code? [J] Is each FK's on-delete/on-update action intentional (no accidental cascade wiping data, no orphan-leaving)? [J] If integrity is app-enforced by necessity, is it enforced on all write paths and justified?
    • Confidence — mixed (cascade-semantics and app-layer-justification are design [J]; the existence of FK/unique/check constraints is deterministic via schema introspection [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Foreign keys make the database itself refuse to create a child record pointing at a parent that doesn't exist, so data can never become orphaned or contradictory — even via a stray script. The cost is a small write-time check and less flexibility for bulk loads; skipping them means integrity lives only in code that any other writer can sidestep.
    • Sources — PostgreSQL docs §5.4 "Constraints" (foreign keys, on-delete actions); Kleppmann Ch. 2 (relational integrity); C. J. Date, Database Design and Relational Theory.

    Idempotent writes¶

    • Good looks like — Write operations that can be retried (network blips, at-least-once queues, client double-submits) are safe to apply more than once without duplicating effect. Achieved via a unique idempotency key / natural unique constraint, INSERT ... ON CONFLICT DO NOTHING/UPDATE (upsert), or a dedup table — so the second delivery is a no-op, not a second charge/order/row. Retried multi-step writes don't leave partial state (combine with transactions). "Exactly once" is treated as at-least-once delivery + idempotent handling, the only honest way to get there.
    • Auditor checks — [J] Can each retriable write be safely repeated (unique key / upsert / dedup), or does a retry duplicate the effect? [D] Is there a unique constraint or idempotency-key column backing the dedup (schema-checkable)? [J] For externally-triggered writes (webhooks, queue consumers, payment callbacks), is replay/duplicate delivery handled? [J] Does a retried multi-step write avoid partial application?
    • Confidence — mixed (whether a path needs idempotency and whether the design achieves it is reasoning [J]; the backing unique constraint/key is deterministically present-or-not [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — Networks and queues retry, and users double-click — so the same "create order" can arrive twice. Idempotent writes make the duplicate a harmless no-op instead of a second order or double charge. The cost is a unique key and a bit of upsert logic; skipping it produces duplicate-record and double-billing bugs that are brutal to trace.
    • Sources — Kleppmann Ch. 11–12 (exactly-once = idempotency + at-least-once; deduplication); Stripe "Idempotent Requests" (stripe.com/docs/api/idempotent_requests — idempotency-key header contract); PostgreSQL INSERT ... ON CONFLICT (postgresql.org/docs/current/sql-insert.html#SQL-ON-CONFLICT).

    Backup before destructive migration¶

    • Good looks like — Any migration that destroys or irreversibly transforms data (drop column/table, type narrowing, mass DELETE/UPDATE, backfill that overwrites) is preceded by a verified, restorable backup or snapshot of the affected data, and ideally a tested rollback. Production runs against a recent point-in-time-recovery baseline; the destructive step is reversible or guarded (e.g. rename-then-drop-later: drop only after a soak period confirms nothing broke). Backups are tested by restoring — an unrestored backup is a hope, not a safeguard. Bulk data ops run in batches with the ability to stop.
    • Auditor checks — [D] Does the destructive migration's runbook/PR require a snapshot/backup before it runs? [J] Is the destructive step reversible or staged (rename/keep then drop after soak) rather than an immediate irreversible drop? [D] Is there evidence backups are restore-tested (not just taken)? [J] Are mass data mutations batched and interruptible?
    • Confidence — mixed (staging/reversibility is design [J]; the backup-before-destructive step and restore-test evidence are checkable in the runbook/process [D] — per-check tags are authoritative).
    • Tradeoff (plain English) — A dropped column or bad bulk update can erase data permanently in seconds, and "undo" doesn't exist for DROP. A verified backup turns a catastrophe into a restore. The cost is a few minutes and some storage before risky changes; skipping it is how teams permanently lose customer data to a one-line migration.
    • Sources — PostgreSQL docs "Backup and Restore" (pg_dump, continuous archiving & point-in-time recovery — postgresql.org/docs/current/backup.html); AWS RDS "Working with snapshots" and PITR (docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_WorkingWithAutomatedBackups.html); Kleppmann Ch. 7 "Transactions" (durability) and Ch. 9 "Consistency and Consensus"; Google SRE Book Ch. 26 "Data Integrity: What You Read Is What You Wrote" (sre.google/sre-book/data-integrity/ — recovery is the goal; test restores, not just backups).

    Authoring rules (the catalog meta-rules — do not delete)¶

    • Additive floor: add dimensions as you discover them; never delete one. This catalog is meant to become "every standard we can think of."
    • Right-size: apply only relevant dimensions per change (KISS/YAGNI); never skip a relevant one. Relevance is a per-change judgment — see README.md.
    • Novel patterns allowed when they add clear value — justify (problem → why existing patterns fall short → benefit) and record in claugentic-DECISIONS.md. Unconventional ≠ wrong.
    • Every dimension carries a Confidence tag so the harness can separate what it proved (deterministic gates) from what it asserts (judgment). Trust the oracle, not the model's word.
